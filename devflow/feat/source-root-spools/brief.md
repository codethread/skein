# Brief: batteries as an ordinary spool via `:skein/source-root` coordinates (u4a24)

Card u4a24. User decision 2026-07-23 (design discussion in session; both parts agreed explicitly). User pre-authorized HITL sign-offs for this run and rebuild/restart of mill/weavers as needed. Revised after cross-vendor review pass (runs zlo0t/b8o2k; findings on step s3lte notes 4jnkj/lugyu).

## Problem

SPEC-004.C50a special-cases `skein.spools.batteries` as the one classpath spool: its source rides `deps.edn :paths`, and the canonical `.skein/init.clj` plus the `DefaultInitCLJ` template `mill init` writes (cli/internal/config/bootstrap.go) activate it through a bare top-level `require` and a guardless `module!`. The stated rationale (spools/README.md "Classpath exception: batteries") is that bootstrap cannot write a batteries coordinate without persisting a machine-specific source-checkout path into shared config: the approved coordinate kinds are closed to `:local/root` (resolved relative to the config dir) and pinned `:git/url`+`:git/sha` (SPEC-004.C42/C48@2/C49@2/C94a), and neither can portably name "the spool shipped with the running weaver's source".

Two consequences:

- Batteries is structurally inconsistent with every other spool: not an approved coordinate, not guarded by `:spools`, invisible in `spools.edn`. Opt-out exists but is incoherent: SPEC-003.C63 lets a workspace mask or replace batteries by editing trusted startup forms, yet source approval and activation are never represented as a `spools.edn` consent edge, and the default template always opts fresh worlds in silently.
- This repo's own `.skein/spools.edn` (and sibling repos') encode the layout assumption that the config dir sits inside the skein source tree, via `{:local/root "../spools/<name>"}` entries.

## Decision

Add a third coordinate kind and make batteries an ordinary approved spool.

1. **New coordinate kind `{:skein/source-root "spools/<name>"}`.** Resolved at sync time against the running weaver's skein source checkout — the same mill-resolved checkout SPEC-004.C50 already binds for blessed namespaces, located by promoting the existing private locator (`skein.core.weaver.runtime`, resource-derived) to a shared, specified authority; never inferred from cwd, config dir, or request state. Fails loudly (TEN-003) when the checkout is unavailable or the resource is not a readable file checkout. Machine-independent and committable. Resolution is confined beneath `<checkout>/spools` by canonical containment after symlink resolution — not merely lexical rejection of `..`/absolute paths. Not batteries-specific: any shipped spool (workflow, text-search, chime, cron, …) becomes consumable this way, coupled to the checkout supplying the running weaver (in-place source edits remain governed by the existing fingerprint/generation rules).
2. **Batteries goes through the approved path.** `spools.edn` gains `skein.spools/batteries {:skein/source-root "spools/batteries"}`; its module declaration becomes a normal `:spools`-guarded `module!`; the bare `require` and guardless module go away. `spools/batteries/src` leaves base `deps.edn :paths` and is added to the `:test` alias `:extra-paths` (`batteries_test.clj` requires it directly) and stays in fmt/lint paths.
3. **`mill init` opts users in by default.** Fresh worlds are seeded with the batteries coordinate in `spools.edn` and a guarded module declaration in `init.clj`. Opt-out becomes coherent and visible: delete the entry. The C50a "zero-config base surface" guarantee is deliberately traded for "mill init opts you in" (explicit user decision; TEN-000@1, no migration).
4. **Sync-diff classification is preserved, sharpened, and tested — not weakened.** A coordinate kind/text rewrite is not itself a refusal when the canonical resolved root is unchanged (today's classifier already compares effective root paths, spool_sync.clj non-additive diff); the independent C44c changed-source-fingerprint, loaded-namespace, and C44f Maven-baseline guards remain fully intact, as does pending-generation behavior. Tests pin the kind-neutral canonical-root identity, and a cutover test proves that in a generation where batteries is still on the immutable launch classpath the synced provider wins and only a fresh generation removes classpath ownership.
5. **Specs are rewritten, not patched around.** SPEC-004.C50a is replaced; C42 admits the kind; C48@2/C49@2 distinguish non-fetching shipped-source resolution from git acquisition; C44's outcome `:kind`/coordinate-field contract carries the third kind; C94a admits it for dependency handling while C94a.1 forbids `:skein/source-root` inside spool-root `deps.edn`; SPEC-003.C63's masking prose updates to the consent-edge model. spools/README's "Classpath exception: batteries" section is replaced with the shipped-spool coordinate story. Generic classpath-ownership machinery (module_refresh classpath classification, base/inherited ledger) stays — C50 blessed namespaces still load from the launch classpath; only batteries-specific exception behavior, comments, and tests go.
6. **Fan-out: no `..` local roots that name shipped skein spools.** This repo's `.skein/spools.edn` migrates its four `../spools/*` entries to `:skein/source-root` (`skein.macros/macros {:local/root "spools/macros"}` stays — genuinely workspace-local). Sibling repos' `.skein` worlds are swept the same way and landed to their mains. Only `..` roots that resolve to spools shipped in the skein checkout can migrate to this kind; any sibling `..` root naming something else is surfaced to the user with a recommendation (git pin or relocation), not silently converted.

Resulting coordinate taxonomy: `:git` = external pinned; `:skein/source-root` = shipped with the running skein; `:local/root` = workspace-local.
