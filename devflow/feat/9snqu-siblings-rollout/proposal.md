# Sibling Spools Mechanical Rollout Proposal

**Document ID:** `PROP-Sbl-001` **Last Updated:** 2026-07-24 **Related RFCs:** None **Related ADRs:** [ADR-003](../../adrs/0003-spool-activation-lifecycle.md) (P5 decision C, P7 conventions) **Related root specs:** [SPEC-003 repl-api](../../specs/repl-api.md), [SPEC-004 daemon-runtime](../../specs/daemon-runtime.md) (C46b) **Prior art:** [PROP-Itr-001](../rrvnn-intree-installer-removal/proposal.md) (in-tree removal, landed `ac5307e`)

## PROP-Sbl-001.P1 Problem

Three sibling spool repos still carry the retired imperative activation path. devflow.spool exports a no-op `install!` metadata shim (`devflow.clj:815`) whose own docstring says to use `contribute`/`reconcile`; its suite still calls it twice, and route publication in tests bypasses the module kernel via `registry/replace-owner!`. kanban.spool exports an `install!` (`kanban.clj:1574`) that duplicates its `contribute`/`reconcile` surface; its two suites activate through it, and both source and test prose still present the installer path as current — including remedies pointing at skein-src's `skein.spools.guild/install!`, deleted in `ac5307e`. agent-harness.spool's src is already clean (OLR F17), but two tests carry generation guards that `requiring-resolve` the deleted `ct.spools.delegation/install!` — always nil now, so the `agent spend` CLI-op path silently stopped being exercised and the agent-failures query test hand-registers what the module contributes.

The version plan drifted from the card (note j2s4g): iteration 10 shipped test-only compat releases devflow v4 / kanban v8 (in-tree activation converted to image-mode `module!`), and agent-harness is at v12 with the 7fg72 cookbook fix already shipped (`75b8a23`). This feature owns each repo's OWN installer surface.

## PROP-Sbl-001.P2 Goals

- **PROP-Sbl-001.G1:** No `defn install!` in devflow.spool or kanban.spool `src/`; agent-harness stays clean. (`install-peering!` is out of scope by recorded decision — see NG2.)
- **PROP-Sbl-001.G2:** Each converted repo exports its base module declaration datum (`<spool>/module`, the ADR-003.P7 `:ns`/`:contribute`/`:reconcile` triple) as the single authored source consumers and fixtures start from.
- **PROP-Sbl-001.G3:** Every suite activates its own spool via `runtime/module!`: devflow and kanban image-mode from the repo classpath; agent-harness through its existing file-backed `activate-module!` helper, which satisfies the module-path goal without a helper-modernization pass (NG5). No test resolves a deleted installer or bypasses the kernel where the module path is the thing under test.
- **PROP-Sbl-001.G4:** A whole-repo prose sweep per converted repo: no source or test docstring, comment, remedy string, or assertion message describes `install!` as current. Gate: `rg 'install!' src test` in devflow.spool and kanban.spool returns only text naming the installer as history (expected: nothing); the specific known survivors are enumerated in S2.
- **PROP-Sbl-001.G5:** Releases: devflow v5 and kanban v9 with `release-exception.md` updated per each repo's existing precedent record (previous marker + peeled sha, proposed marker, removed name, known consumers, exact expected compat-alarm failures with no unrelated failure accepted, decision, rollback); agent-harness v13 test-only (no break record needed). Annotated tags **pushed**, peeled shas (`git ls-remote` `refs/tags/vN^{}`) recorded on the card.

## PROP-Sbl-001.P3 Non-goals

- **PROP-Sbl-001.NG1:** No consuming workspace pin bumps — the cutover feature (rtnfv) owns them. The skein-src `test/skein/config_ops_test.clj` conversion is NOT staged-for-later: S5 converts it v8-compatibly so it lands with this feature and stays green across the cutover.
- **PROP-Sbl-001.NG2:** `install-peering!` stays imperative: kanban.spool commit 5e7cb5c records peering activation as a dedicated opt-in adapter module in consuming config; it is not a `defn install!` and the epic DONE-WHEN does not cover it. Its prereq remedies and docstrings are reworded (G4/S2), not removed.
- **PROP-Sbl-001.NG3:** dresser.spool — feature kst0n (parallel, real redesign).
- **PROP-Sbl-001.NG4:** notebook.spool and notes need no code change (grep-verified clean of `install!`); recorded as a card note, no releases.
- **PROP-Sbl-001.NG5:** agent-harness's file-based `activate-module!` test helper is not migrated to `:load :image` wholesale; the delegation conversion uses the existing helper. A helper-modernization pass without a driving defect is churn.

## PROP-Sbl-001.P4 Proposed scope

- **PROP-Sbl-001.S1 (devflow.spool → v5):** Export `devflow/module`. Delete `install!`. Convert `publish-devflow-routes!` in `devflow_test.clj` to declare the `:devflow` module via `runtime/module!` (image mode, `:after [:workflow]`), so the route-repoint and route-deletion tests exercise replacement and deletion-by-omission through the actual kernel instead of `registry/replace-owner!`; `with-runtime` activates workflow then devflow. The `production-return-coverage-is-derived-from-devflow-provenance` test is **deleted with rationale**: devflow contributes only workflow constructors, never `:ops` (`devflow.clj:769-779`), so the test's provenance-filtered op set is empty and every assertion is vacuous — real devflow op returns are registered and covered by the consuming workspace (skein-src `config_ops_test.clj`'s owner-return-coverage over `'config` provenance). The dependency-sentinel test drops the installer-metadata assertion and keeps the direct fn check.
- **PROP-Sbl-001.S2 (kanban.spool → v9):** Export `kanban/module`. Delete `install!` (behavior audit done, review-verified: everything it registers is in `contribute`; vocab and spool-state live in `reconcile` — the module path is a superset; tracker binding was never installed). Convert `with-kanban` (`kanban_test.clj`) and the four `kanban_peering_test.clj` call sites to module activation. Whole-repo prose sweep (G4), enumerated survivors: `kanban.clj:869-878` (`set-tracker!` docstring describing installer behavior), `kanban.clj:1595-1602` (`install-peering!` docstring prescribing both deleted installers), `peering.clj:4-6` and `:721-722` (ns/fn docstrings), `peering.clj:~693/697` (prereq remedy strings), `kanban_test.clj:45,101` (fixture docstring + assertion prose), `kanban_peering_test.clj:68,85` (remedy regexes) and `:103-104` (test prose). Remedies now prescribe module activation (`runtime/module!` with the exported datum or the consuming world's declaration).
- **PROP-Sbl-001.S3 (agent-harness.spool → v13):** Main fast-forwarded to origin (`c7d2266`, done). Replace `delegation-install` (`agent_run_test.clj`) and the guard in `install-agent-failures-query!` (`subagent_test.clj`) with delegation module activation via the repo's `activate-module!` helper, making the spend test unconditionally exercise the `agent spend` op and the failures-query test consume the contributed query. Tidy stale installer wording in `bench_test.clj:793-794` assertion messages, `bench_metrics_test.clj`, `delegation_test.clj` testing strings, `subagent_test.clj:624` comment.
- **PROP-Sbl-001.S4 (releases):** Per repo: suite green against skein-src main (5bae1: `:test` aliases hard-code `../skein-src`; keep main checked out and green there), `bin/compat-alarm` vs previous marker. For v5/v9 the alarm is **expected to fail exactly at the archived tests that resolve the deleted installer** — no unrelated failure accepted; `release-exception.md` updated per the repo's precedent shape before tagging. Annotated tags pushed to origin; peeled shas recorded on the card. Tags are immutable; mistakes ride the next marker.
- **PROP-Sbl-001.S5 (skein-src config_ops_test conversion):** `test/skein/config_ops_test.clj:155`'s `((requiring-resolve 'ct.spools.kanban/install!))` converts to a literal image-mode declaration using only symbols that exist at the current v8 pin (`{:ns 'ct.spools.kanban :load :image :contribute 'ct.spools.kanban/contribute :reconcile 'ct.spools.kanban/reconcile}` after requiring the ns), verified green against the pinned v8 before landing. Because the literal triple is valid at v8 and v9 alike, this lands WITH this feature on this branch — no unlandable staged commit, nothing handed to rtnfv beyond the ordinary pin bump.
