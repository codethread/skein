## DELTA-Dsp-002.P1 Summary

This delta owns the refresh-coordinator side of Phase A of the `def spool` convention (PROP-Dsp-001): where a module's entry points come from, how image mode resolves them, and what the coordinator retains so removal teardown keeps working. The declaration grammar, the `::spool` spec, and the repository lint gate are in the REPL API delta (`DELTA-Dsp-001`).

The change is deliberately narrow. Only the source of a module's `:contribute`/`:reconcile` pair changes, plus one retention seam so removal-by-omission still has a reconciler to run. Every other refresh, reconcile, and runtime invariant is unchanged (SPEC-004.C44–C46, C46a, C46b, C70, C94–C96).

## DELTA-Dsp-002.P2 Contract changes

- **DELTA-Dsp-002.CC1 (convention resolution at every module evaluation — amends SPEC-004.C46):** The refresh coordinator resolves a module's `:contribute`/`:reconcile` from the public `spool` var in the module's loaded namespace at **every module evaluation**, not only after a fresh source load — so the `:unchanged` fast path, classpath bindings, targeted refreshes, and namespaces touched by `reload-code!` all resolve identically. For an `:ns` target the var is resolved from the declared namespace; for a `:file` target the module must declare exactly one `ns` form and the var is resolved there, and a file with no `ns` declaration keeps today's behavior (authoring forms only, no convention lookup); for a `:load :image` target the var is resolved from the JVM image (CC3). Unqualified `spool` symbols are qualified against the declaring namespace at resolution time, so the published declaration stays fully qualified printable data and SPEC-004.C45's printable-declaration invariant holds at the publication boundary. The `::spool` shape and author-facing validation are `DELTA-Dsp-001.CC2`.

- **DELTA-Dsp-002.CC2 (Phase A transitional per-key precedence — amends SPEC-004.C46):** During Phase A the grammar still accepts `:contribute`/`:reconcile` (`DELTA-Dsp-001.CC1`). Resolution fills only **absent** entry-point fields from the `spool` var; an explicitly declared value wins **per key**, silently, and is documented as transitional. A complete legacy explicit declaration remains valid when its target namespace carries no `spool` var — the coordinator consults the var only for fields the declaration leaves absent. Phase C removes the keys and this transitional merge with them. This is precedence, not conflict: the pinned sibling suites still declare explicit entry-point keys for in-tree namespaces during the window, and a hard conflict would fail those suites the moment those namespaces gain `spool` vars.

- **DELTA-Dsp-002.CC3 (image mode resolves `spool`, `:contribute` no longer required — amends SPEC-004.C45/C46):** A `:load :image` module no longer requires an explicit `:contribute`; image evaluation resolves `<ns>/spool` from the already-loaded JVM image and fills only fields absent from a Phase A explicit declaration. SPEC-004.C45's declaration-time refusal of a `:load :image` declaration missing `:contribute` (the `:required :contribute` refusal) is deleted; the `:file`-target refusal (`:allowed [:ns]`) and the `:load`-value check (`:allowed #{:image}`) stay declaration-time refusals. An unloaded image namespace fails at evaluation. A loaded namespace with no public `spool` var may still use an explicit Phase A `:contribute`; when neither source supplies contribution, the module is a loud evaluation-time `:failed` outcome on the same per-module evaluation-failure channel, never a top-level `:refused` and never a throw out of `module!`, with error data naming `:module/key`, the offending `:ns`, and `:load :image`. The image outcome still states `:source/status :image` and carries no source stamp, in `plan` and `status` alike. This resolves ADR-003.P9's first revisit trigger (explicit `:contribute` as image-mode friction) without relaxing any loud failure.

- **DELTA-Dsp-002.CC4 (retained last-good resolved entry points and status/plan exposure — extends SPEC-004.C46/C46b):** The coordinator persists each module's **last successfully resolved entry-point set** in runtime state. A module removed by omission is never re-evaluated — its source is not loaded on the removal path — so the coordinator reconciles the `:removed` teardown (SPEC-004.C46b) through that retained set rather than a freshly resolved one. `status`, `plan`, and refresh results expose the resolved entry points per module as a resolved field alongside the authored declaration; today's view shows only the authored graph. The retained set is last-good: after a failed evaluation the previously resolved set is retained and the failure is reported, and it persists across `reload-code!`. Without this seam, dropping `:reconcile` from declarations would make removal teardown silently disappear — the parity-bug class ADR-003.P6 (SPEC-004.C46b) exists to prevent.

- **DELTA-Dsp-002.CC5 (loud enforcement, enumerated — extends SPEC-004.C45/C46):** Evaluation-time failures carry named-field, actionable errors for: a `spool` value that is not a map, has unknown keys, or has neither entry point; an entry-point value that is not a symbol (ADR-002.O1); a symbol that does not resolve, or resolves to a var whose **root value** is not a fn — the check derefs the var and tests the held value, because a Clojure Var is itself `ifn?` regardless of what it holds, so an `ifn?`-on-the-var test cannot give the promised error; and a `spool` var carrying `:contribute` while the same source load also collected authoring forms, which is a loud conflict forcing the author to choose. A `:reconcile`-only `spool` var **composes** with collected authoring forms and is not a conflict — forms-as-contribution plus a reconciler is valid production behavior today and must survive. An ns module with neither a `spool` var nor collected forms stays legal (an empty contribution is a legitimate pattern, not an error). Stale `spool` vars left in a namespace after an in-place reload keep exact parity with today's symbol resolution; no var-cleanup machinery is added, and the existing pickup ladder is the documented remedy in both worlds.

- **DELTA-Dsp-002.CC6 (invariants unchanged — restates SPEC-004.C44–C46, C46a, C46b, C70):** Staging versus refresh collection, full-graph recollection from startup files, source stamps and the `:unchanged` fast path, residual/conflict/remedy reporting, root guards, dependency ordering, atomic owner-partition publication, effective-registry retraction (SPEC-004.C46a), and the reconcile status-branching contract (SPEC-004.C46b, ADR-003.P6 decision D) are all unchanged. Only where a module's entry-point pair comes from changes, plus the retention seam CC4 adds. No generic teardown or lifecycle callback enters the publication kernel; teardown stays the reconcile fn's `:removed` branch (the DELTA-OlrRepl-001.CC13 boundary).

## DELTA-Dsp-002.P3 Design decisions

- **DELTA-Dsp-002.D1 (image mode's one declaration→evaluation locality move is accepted):** For ordinary modules, declared entry-point symbols already resolve at evaluation time after source load, so convention resolution changes nothing about where failures land. Image mode is the sole exception: its missing-entry-point refusal fires at declaration time today because the declaration requires `:contribute`. Resolving `spool` from the image moves that one refusal to an evaluation-time `:failed` outcome, because the namespace cannot be inspected before load/lookup. **Rationale:** the check has no honest declaration-time form once the entry point lives in the image. **Rejected:** relaxing the missing-`spool` case to a silent default (PROP-Dsp-001.NG4; ADR-003.P9, "do not revisit by relaxing the loud failures").

- **DELTA-Dsp-002.D2 (retain the resolved set rather than re-load removed source):** Removal-by-omission never re-evaluates a removed module, and the coordinator already recovers a removed module's reconciler from previously stored declaration state. **Rationale:** retaining the last-good resolved entry-point set matches that existing recovery path once declarations stop carrying `:reconcile`. **Rejected:** re-loading a removed module's source to re-resolve its `spool` var (contradicts removal-by-omission), or dropping teardown (the ADR-003.P6 parity-bug class).

- **DELTA-Dsp-002.D3 (per-key precedence over a hard conflict during the window):** See CC2. **Rationale:** the pinned sibling suites keep explicit entry-point keys for in-tree namespaces until Phase B, so precedence lets a `spool` var and an explicit key coexist through the transitional window. **Rejected:** a hard "declared key + `spool` var" conflict in Phase A, which fails the pinned suite gate before Phase B can land.

- **DELTA-Dsp-002.D4 (the forms conflict is scoped to `:contribute` only):** A `spool` var whose `:contribute` is present alongside collected authoring forms is a loud conflict; a `:reconcile`-only var composes with forms. **Rationale:** forms-as-contribution plus a reconciler is valid production behavior today and must survive the conversion. **Rejected:** a blanket forms-versus-`spool` conflict that would break that combination.
