# Large-attribute Scaling (assess) Proposal

**Document ID:** `PROP-LargeAttrScaling-001` **Last Updated:** 2026-07-11 **Related RFCs:** None (no RFC covers attribute storage representation; the adjacent [RFC-002 Task Query DSL](../../rfcs/2026-06-24-task-query-dsl.md) defines the query language the assessed paths run over, unchanged here) **Related root specs:** [Strand Model](../../specs/strand-model.md) (P4 Attributes, P8 Persistence, P9 Query fields), [Weaver Runtime](../../specs/daemon-runtime.md) (P3a storage model), [Alpha Surface](../../specs/alpha-surface.md) — as context for the assessment; this spike stages **no** spec deltas (`PROP-LargeAttrScaling-001.NG2`). **Builds on:** [attr-scaling-ship-now](../attr-scaling-ship-now/proposal.md) (`PROP-AttrScalingShipNow-001`: L0a pragmas + L1 lean reads; L2 overflow side table deferred, then obsoleted) and [eav-attr-storage](../eav-attr-storage/proposal.md) (`PROP-EavAttrStorage-001`: EAV one-row-per-`(strand_id, key)` storage, JSON `TEXT` scalar values with a `json_valid` CHECK, an `archived` flag with `WHERE archived = 0` partial hot indexes, and the migrate cutover — both SHIPPED). This proposal assesses the storage baseline those two features left in place; it does not change it.

## PROP-LargeAttrScaling-001.P1 Problem

Kanban card `kbcjt` ("Assess large-attr scaling: spike offload/schema options under synthetic load") predates both shipped features above and its original question — *should attributes offload to a side table or a different schema?* — is largely answered: `PROP-EavAttrStorage-001.P1` adopted EAV-with-archive and rejected transparent hot offload (the deferred L2 side table), an artifacts second concept, event sourcing, and fixed attribute columns. The card was rescoped with coordinator sign-off (card note `w9v7t`) to close two gaps that survive that decision:

- **F1 — no committed shipped-baseline large-attr harness.** The benchmark gates that blocked `PROP-EavAttrStorage-001` merge (`EAS-PLAN-001.P7`: write-amp `≥5×` on ≥16 KiB payloads, 250k-strand filtered scans within the document envelope, `list`-of-500 assembly `≤2×`, `ready` `≤1.7×` — measured `1.69×`) were adapted from the card-`bvb0g` spike harnesses held under `/tmp/claude/attr-scaling-ship-now-brief/`. Those original harnesses were `/tmp`-ephemeral and were never committed. `dev/skein/eav_benchmark.clj` is committed (the `eav-attr-storage` merge gate) and reproduces the gate numbers in hand-SQL, but it is only the document-vs-EAV migration-ratio comparison — it exercises no shipped `skein.core.*` code and covers none of the `F2` large-value/archived-row regimes below. The committed, shipped-baseline large-attr harness this proposal calls for is still missing.

- **F2 — residual full-fidelity paths were reasoned about, not measured against the shipped baseline.** The EAV design accepted several "correct-but-slower on cold data" paths (TEN-003 forbids *wrong*, not *slow*) and pushed one cost to a follow-up card, without a committed measurement of any of them on the shipped row schema:
  - **Full-fidelity point reads** (`strand-row-by-id`, `src/skein/core/db.clj:371-375`) transparently include archived rows and reassemble the whole attribute map from N rows per strand.
  - **Payload inlining.** Payload bodies still inline directly into attribute-row values with no file offload (`:payload/*` / `:stdin` references resolve into attribute values — `src/skein/api/cli/alpha.clj:324-357`), so multi-KB..MB values live as single JSON `TEXT` rows.
  - **Text-search LIKE scans.** The text-search spool (`spools/text-search/src/skein/spools/text_search.clj`) runs `LIKE` over the physical tables and can include archived rows when `--archived`/`:archived?` is set (`text_search.clj:110-119`, `:139-157`; archived rows excluded by default); its cost against large inlined values and high archived-row volumes is unmeasured.
  - **Assembly cost on large lists.** `ready`/`list` batch-assemble the attribute map, partitioning the *strand-ids* into batches of 30000 (`attribute-assembly-batch-size`) and `json_group_object`-aggregating the per-strand attribute rows — `src/skein/core/db.clj:340-357`; lean list paths `all-strands-lean` `db.clj:1261-1295`, `ready-strands-lean` `db.clj:1341-1408`), the residual cost EAV's `1.69×` `ready` result left to bounded-frontier card `ncso4` (now done). Whether large inlined values or archived volumes move that number is unmeasured.

  The 1024-byte omission floor (`attribute-value-sql`, `src/skein/core/db.clj:333-338`) is injected into the shared assembly helper (`attribute-json-by-strand-id`, `db.clj:340-357`), so lean `list`/`ready` assembly already omits `>1024`-byte hot values. It does not reach the full-fidelity point-read or text-search paths (which resolve whole values, and on point reads archived rows too), and it caps value payload but not the per-strand row aggregation — N attribute rows are still scanned and grouped per strand regardless of value size. The residual concerns this spike measures are therefore the row-aggregation cost on large lists, archived-row inclusion on the full-fidelity point-read (and text-search when `:archived?` is set), and the full-fidelity/text-search value materialization the lean floor does not cover. It assesses whether any residual read-path mitigation — consistent with the settled EAV+archive storage and TEN-007 — is worth a follow-up feature.

## PROP-LargeAttrScaling-001.P2 Goals

- **PROP-LargeAttrScaling-001.G1 — A committed, reusable harness.** Deliver a synthetic large-attr load harness that lives in the repo, runs on disposable workspaces, reproduces the shipped EAV bench gates (`EAS-PLAN-001.BG1`–`BG4`), and adds the large-value scenarios `F2` left unmeasured. It replaces the lost `/tmp` harnesses as the durable baseline for any future attribute-storage change.
- **PROP-LargeAttrScaling-001.G2 — Numbers on the shipped baseline.** Run the harness against the current shipped storage and commit an assessment report carrying the measured numbers, so the accepted gates become reproducible and the residual-path costs become concrete.
- **PROP-LargeAttrScaling-001.G3 — An evidence-backed options verdict.** Assess residual read-path mitigations for the full-fidelity paths (`F2`) and large-list assembly — options that stay within the settled EAV+archive storage (`PROP-EavAttrStorage-001.P1`, `PROP-LargeAttrScaling-001.NG1`), not a reopening of the rejected side-table/schema/offload question — and either recommend a specific follow-up feature or explicitly recommend no action, with the measured evidence behind the call, respecting TEN-007 (no proposed option may make a consumer above `skein.core.*` depend on the physical shape of attribute storage).

## PROP-LargeAttrScaling-001.P3 Non-goals

- **PROP-LargeAttrScaling-001.NG1:** No relitigating the settled storage direction. The alternatives rejected in `PROP-EavAttrStorage-001.P1` (transparent hot offload / L2 side table, artifacts as a second concept, event sourcing, fixed attribute columns) stay rejected; this spike does not reopen them. If measurement surfaces a genuinely new option it is recorded as a follow-up recommendation (`PROP-LargeAttrScaling-001.G3`), not built or re-argued here.
- **PROP-LargeAttrScaling-001.NG2:** No schema migration and no storage-representation change in this spike. It is assessment only — it stages no spec deltas and touches no `skein.core.*` storage code. Any change it recommends is a future feature with its own proposal and its own migrate/cutover discipline (TEN-000@1, TEN-003).
- **PROP-LargeAttrScaling-001.NG3:** No new public CLI or agent surface (TEN-004, TEN-006). The harness is a dev/test tool, not a shipped command; its output is a committed report, not a runtime feature.
- **PROP-LargeAttrScaling-001.NG4:** No archive **policy** work. What to archive and when stays userland (`PROP-EavAttrStorage-001.NG4`); the harness may *populate* archived rows to measure their effect, but proposes no gleaner or policy.
- **PROP-LargeAttrScaling-001.NG5:** No implementation strategy, harness internals, phase breakdown, or task slicing here — those belong in the feature plan.

## PROP-LargeAttrScaling-001.P4 Proposed scope

- **PROP-LargeAttrScaling-001.S1 (Committed synthetic large-attr harness):** A reusable synthetic large-attr load harness that lives in the repo as a **dev/test tool**, not a public CLI or agent surface (`PROP-LargeAttrScaling-001.NG3`). It reproduces the four shipped EAV bench gates (write-amp on payload-carrying strands, 250k-strand filtered scans, `list`-of-500 assembly, `ready`-view latency; `EAS-PLAN-001.BG1`–`BG4`) **and** extends coverage to the large-value and archived-row regimes `F2` left unmeasured — values around the lean-read floor, inlined payload bodies up to MB scale, text-search `LIKE` scans, and populated archived-row volumes. It replaces the lost `/tmp` harnesses as the durable baseline.
- **PROP-LargeAttrScaling-001.S2 (Baseline run + committed report):** A run of `S1` against the current shipped storage baseline, producing a committed assessment report in this feature folder that records the reproduced gate numbers and the new large-value scenario numbers, so the accepted gates and the residual-path costs are both durable and re-runnable.
- **PROP-LargeAttrScaling-001.S3 (Residual options assessment):** An evidence-backed assessment, in the same report, of whether any residual read-path mitigation earns a follow-up for the full-fidelity paths (point reads over archived rows, text-search `LIKE` scans, payload inlining) and large-list assembly cost (relates to card `ncso4`). Options are bounded to those that stay within the settled EAV+archive storage — not a reopening of the rejected side-table/schema/offload question (`PROP-LargeAttrScaling-001.NG1`). It ends in a concrete verdict — a named follow-up feature or an explicit no-action recommendation — grounded in the `S2` numbers and constrained by TEN-007.

## PROP-LargeAttrScaling-001.P5 Open questions

- **PROP-LargeAttrScaling-001.Q1:** Harness home and shape — a committed `test/`-tier namespace, a dedicated dev alias, or both (a fast smoke slice for CI plus a slow full-scale run gated by an env flag), resolved in the plan against the tiered-validation discipline.
- **PROP-LargeAttrScaling-001.Q2:** What "shipped baseline" pins to for reproducibility — a named `main` commit and a documented synthetic-world seed (250k strands, payload profile) so a later re-run is comparable.
- **PROP-LargeAttrScaling-001.Q3:** Whether any large-value scenario becomes a **merge-blocking gate** or stays informational. Since this spike ships no storage change (`PROP-LargeAttrScaling-001.NG2`), the default is informational; a follow-up feature would own any new gate.
- **PROP-LargeAttrScaling-001.Q4:** Whether text-search seeing archived rows (`spools/text-search`) is a correctness concern worth a follow-up in its own right, or purely a cost question, decided by the `S2` measurement.
