## TASK-007.P2 Implementation notes

- Create a temporary external-style fixture outside the Skein checkout, preferably under a short path such as `/tmp/skein-ext-fixture-<suffix>` to avoid Unix socket path limits.
- The fixture should have its own `deps.edn`, `src`, and `test` directories.
- Its `deps.edn` should add this Skein checkout as a test/dev `:local/root` dependency, not rely on a globally installed Skein or the user's default config-dir.
- Include tests that prove:
  - pure library code tests run without weaver assumptions
  - the author test JVM can require Skein namespaces, including `skein.test.alpha`
  - `skein.test.alpha` can start a disposable weaver world for the fixture
  - the fixture spool is approved via generated `spools.edn`
  - weaver-side `skein.api.runtime.alpha/sync!` and `use!` (explicit runtime argument) can load/activate the fixture spool
  - both `:storage :sqlite-file` and `:storage :sqlite-memory` are exercised, or document why one mode is intentionally not used in the fixture
- Keep the fixture temporary or generated by a test/helper script; do not commit a large vendored test project unless the implementation benefits clearly outweigh the maintenance cost.
- If docs examples drift from the working fixture, update `docs/library-authoring.md`.

## TASK-007.P3 Done when

- A separate external-style fixture successfully runs tests against this Skein checkout.
- The fixture proves `skein.test.alpha` is consumable from `src/skein/test/alpha.clj` via tools.deps local-root.
- The fixture proves real weaver spool activation through `spools.edn`, `sync!`, and `use!`.
- The exact command and result are recorded in the plan Developer Notes or a committed focused test script if one is added.
