## PLAN-Ooc-001.P1 Goal and scope

Deliver RFC-019: collapse the dual dispatch tiers so every weaver-facing command is a registered op at the CLI root, `strand` becomes a pure invoke-envelope dispatcher, `mill` absorbs the no-weaver commands, and the shipped command surface moves to `skein.spools.batteries` built on a new blessed parser `skein.api.cli.alpha`. Hard cutover per TEN-000; contracts in the three feature spec deltas.

## PLAN-Ooc-001.P2 Approach

- **PLAN-Ooc-001.A1:** Build weaver-side foundations first (registry metadata, parser, help), then batteries, then the socket protocol, then the Go cutover, then repo-wide prefix removal. Phases 1–2 are additive and land while the old CLI still works: batteries ops are reachable through the existing `strand op` path, so each phase is independently testable against a disposable workspace before the breaking cutover in phases 3–4.
- **PLAN-Ooc-001.A2:** Registry accretion: `register-op!` gains an op metadata map (doc, arg-spec, `stream?`, deadline class, hook-gating class) and switches from silent-replace to loud collision; explicit `replace-op!` is the override. Safe with `reload!` because reload clears registries before re-running init files (SPEC-004.C46). The registry records provenance (registering namespace) itself. Handler context accretes envelope fields (`:op/payloads`, `:op/cwd`, `:op/worktree-root`, `:op/git-common-dir`, `:op/timeout`) alongside the existing `:op/name`/`:op/argv`/`:op/runtime` keys; parsed args are provided when the op declares an arg-spec.
- **PLAN-Ooc-001.A3:** `skein.api.cli.alpha` owns argv parsing and payload-reference resolution (whole-value `:stdin` / `:payload/<name>`, the two loud rules from SPEC-003-D003.C2) and renders per-op help from the same arg-spec. Core `help` op upgrades from the current `op-help-handler` to a full registry projection.
- **PLAN-Ooc-001.A4:** Socket cutover replaces the allowlist + `dispatch` case in `skein.core.weaver.socket` with `invoke` (envelope → `op!`) plus minimal `status` health/identity; `stop` becomes signal-driven shutdown owned by mill supervision. Response framing gains the header/lines/terminator stream shape; mill's forward path must relay stream frames, not buffer single responses.
- **PLAN-Ooc-001.A5:** Go cutover: `cli/internal/command` Cobra tree is deleted; `strand` becomes dispatcher-flag parsing + envelope assembly + NDJSON relay (`--dry-run` prints the envelope locally); `mill` gains `init` and `weaver *` subcommands by moving the existing implementations behind new command names.
- **PLAN-Ooc-001.A6:** Repo cutover: local spools (`shuttle`, `agents`, `treadle`, `chime`, `backlog`) and any scripts/docs that shell `strand op <name>` or old builtins switch to the new invocations; smoke test rewrites to the new surface; CLAUDE.md and `.agents/skills/strand/SKILL.md` update. The external `codethread/devflow.spool` needs its own update and a `spools.edn` sha bump; until then a `spools.local.edn` override on the feature branch keeps development unblocked.

### TASK-Ooc-002 blessed parser skein.api.cli.alpha (2026-07-04)

- **Arg-spec grammar (owned design).** Chose a single `:type` axis with four values — `:string`/`:int` consume the following token, `:boolean` is a presence form (no value consumed), and `:map` accumulates `--flag key=value` tokens into a `{k v}` string→string map. MI1 lists map flags separately from typed flags, but folding "map" into `:type` keeps the grammar minimal EDN data with no functions; map flags are inherently repeatable (each occurrence merges) so they ignore `:repeat?`, which only applies to `:string`/`:int`. Positionals are an ordered vector keyed by `:name`; the trailing one may set `:variadic? true` to collect remaining tokens into a vector. Parsed result is keyword-arg-name → value.
- **Payload map keys are strings.** `resolve-payloads` normalizes the envelope payloads map to string keys via `(name k)` before matching, so `:stdin` → payload name `"stdin"` and `:payload/<name>` → `<name>`. Whole-value matching only: `= ":stdin"` or `starts-with? ":payload/"` (the remainder is the name by construction). Both loud rules enforced: a ref with no payload throws `:missing-payload` (ex-data lists `:available`); any payload no ref consumed throws `:unused-payloads` (ex-data lists `:unused`). Consumed tracking uses a function-local atom, not module-level state.
- **`:parse` applies post-resolution to literal or payload strings.** MI4's `:json`/`:jsonl` runs after payload resolution and applies to whatever string the arg holds (literal inline JSON or a resolved payload), recursing over repeat-vectors and map-flag values. `:jsonl` skips blank lines and reports the 1-based line number in `:malformed-jsonl` ex-data.
- **Error shape.** Every failure is an `ex-info` carrying `::error true`, a `:reason` keyword (`:unknown-flag`, `:missing-required`, `:type-violation`, `:duplicate-flag`, `:malformed-kv`, `:missing-value`, `:trailing-tokens`, `:missing-payload`, `:unused-payloads`, `:malformed-json`, `:malformed-jsonl`), the `:op` from the arg-spec, and the offending token/flag. Task 3 (registry/help integration) and userland spec/malli layers can dispatch on `:reason`. Added `:missing-value` beyond MI2's enumerated set as the loud failure for a value flag with no following token.
- **Test runner registration.** `skein.api.cli.alpha-test` is a pure namespace, so it was added to `parallel-namespaces` in `test/skein/test_runner.clj` (no add-libs/JVM-global state). Full `clojure -M:test` green (417 tests / 2128 assertions).
- **No RFC/delta conflicts.** SPEC-003-D003.C1/C2 matched implementation as built; `explain` returns JSON-safe data (strings/bools/nil/vectors/maps) ready for the task-3 `help <op>` projection.

### TASK-Ooc-003 parser-op integration + core help projection (2026-07-04)

- **`op!` parses eagerly during ctx construction.** The `(some? arg-spec) (assoc :op/args (cli/parse arg-spec argv payloads))` branch runs while building the ctx map, i.e. before `with-spool-classloader` calls the handler, so a parse failure throws the parser's structured `ex-info` and the handler never runs — MI1's short-circuit falls out of ordering, no explicit guard needed. Raw-envelope ops (no `:arg-spec`) get no `:op/args` key at all (not a nil), so a handler can distinguish "no parser ran" from "parsed to nil", and they still carry the raw `:op/payloads` map untouched. The parser is pure, so it runs outside the spool classloader deliberately.
- **Help projection shape.** Listing is `{:ops [<summary> ...]}` where each summary is `{:name :provenance :stream? :deadline-class :hook-class}` plus `:doc` when present, ordered by `ops` (canonical-name sort). Detail is that same summary `merge`d with either `{:arg-spec (cli/explain ...)}` or the `{:raw-envelope true}` marker. Kept provenance/classes as native symbols/keywords (matching the old `:registered (ops ...)` payload and the rest of the registry API); the socket JSON layer stringifies them, and `explain` already emits JSON-safe strings for the arg-spec sub-tree.
- **String→symbol at the help detail boundary.** The parsed `op` positional arrives as a raw string, but `resolve-op`→`canonical-op-name`→`query/canonical-query-name` only accepts simple symbols/keywords (rejects strings). The handler symbolizes with `(symbol op-name)` before `resolve-op`, which preserves the existing loud "Operation not found" error carrying `:available` for unknown names (MI2). No new op-name lookup variant was added.
- **DW2 (live disposable-workspace CLI check) substituted by socket-protocol coverage.** This delegated shuttle run is contractually barred from starting mills/weavers, so the `strand op help` / `strand op help help` live end-to-end check against a locally built current-main CLI was not executed. Instead `json-socket-op-dispatch` now drives both the listing (`args []` → `result.ops`) and the detail (`args ["help"]` → `result.arg-spec.op`, no `raw-envelope`) through the real JSON socket handler — the same `op!`→help path the Go dispatcher relays (Go relay unchanged; OS1). Follow-up: exercise the actual built-binary path in the PH2/smoke work once weaver-start is permitted, per plan V2/V3.

### TASK-Ooc-004 batteries core strand ops (2026-07-04)

- **Classpath spool, not a coordinate.** `skein.spools.batteries` lives under `spools/src` (already on the base `:paths`), so like `ephemeral`/`workflow`/`guild` it needs no `spools.edn` approval or `sync!` — tests just `require` it and call `(batteries/activate! rt)`. `activate!` has a no-arg `use!`-conventional arity (`(current/runtime)`) plus an explicit-runtime arity for tests/trusted callers. Op handlers never touch the published singleton: they read `(:op/runtime ctx)` from the invocation context, honouring the runtime-publication discipline.
- **Duplicate `--attr` loudness vs. the parser's `:map` type (discovery/follow-up).** Old C6e requires duplicate keys *within* one `--attr` priority to fail loudly, but the blessed parser's `:map` flag (TASK-Ooc-002) silently collapses duplicate keys via `assoc`, and that collapse is irreversible after parse. To honour C6e without diverging, `add`/`update` enforce the rule in the handler by recovering the flag keys from the raw `:op/argv` (`check-attr-duplicates!`), which is safe because the parser guarantees each `--attr` is followed by a well-formed `key=value` token. This is a genuine limitation of the shared parser, not a spec conflict: a future parser enhancement — a duplicate-key-loud `:map` variant (or a `:map` option) — would let batteries drop the raw-argv precheck. `--param` keys intentionally still last-wins-collapse, matching the old CLI's `parseKV` (non-dedup) behaviour, so only `--attr` carries the precheck. Payload-ref *values* inside `--attr`/`--attributes` are resolved by the parser (map `:map` values and the single `:attributes` string are whole-value refs), replacing old `--attr-file`/`--attr-stdin`/`--attributes-stdin`.
- **JSON-shape parity is structural, via the same `skein.api.*` calls.** Each op delegates to exactly the `skein.api.weaver.alpha` fn the socket `dispatch` case uses (`add`/`update`/`show`/`supersede`/`burn-by-ids`/`list`/`ready`/`subgraph`), with the named-query param validation and list state-overlay reproduced from the socket's private `query-params`/`dispatch-query`. Mutating ops pass a `{:request/source :json-socket :request/operation <kw>}` context so hooks/events see equivalent data (MI5). `subgraph` returns the socket's string-keyed `{"root_ids" ... "strands" ... "edges" ...}` shape verbatim; it takes a single `<root-id>` positional + `--relation` per MI4 (the socket op still accepts a `root_ids` vector, which batteries wraps as `[root-id]`).
- **DW2 (live disposable-workspace CLI check) substituted by `op!`-path coverage.** This delegated shuttle run is contractually barred from starting mills/weavers, so the `strand op add/list/ready` live end-to-end check against a built CLI was not run here. Instead `skein.spools.batteries-test` drives every op through `skein.api.weaver.alpha/op!` — the exact entrypoint the socket `op` dispatch relays to (Go relay unchanged) — including the envelope-arity payload path for `--attr body=:stdin` and `--attributes :payload/attrs`. Follow-up: exercise the built-binary `strand op` path in PH2/smoke once weaver-start is permitted (plan V2/V3).
- **No RFC/delta conflicts.** SPEC-002-D004.R2 flag-semantics mapping and old SPEC-002.C6–C11 matched implementation reality; the only friction was the parser dup-collapse noted above, resolved within batteries rather than by diverging.

### TASK-Ooc-006 socket invoke/stream/signal cutover (2026-07-04)

- **Socket collapsed to `invoke` + `status`; `handle-request` became `handle-request!`.** `skein.core.weaver.socket` lost `allowed-operations`, `argument-error`'s per-command shape table, `dispatch`, `dispatch-query`/`query-params`/`handle-name`, the CRUD/query/weave/subgraph cases, and the socket `stop` case. The request frame is unchanged (SPEC-004.C23). `invoke` validates only the envelope shape (`name` non-blank string, `argv` string vector, `payloads` string→string map required; `cwd`/`worktree_root`/`git_common_dir`/`workspace`/`timeout`/`client` optional and typed) — argument validation now belongs to ops via the parser. `handle-request` was replaced by `handle-request!`, which writes through a `write-frame!` callback (one flushed NDJSON line per frame) so a stream op can emit many frames; `socket/start!` dropped its `stop-fn` parameter and `runtime/start!` was updated to the 2-arg call.
- **Client identity is diagnostic-only.** `workspace` and `client{pid,version}` ride the invoke arguments and are visible to payload hooks in `:request/args` (the whole decoded arguments map is passed) but are *not* threaded into op handler context — `invoke-envelope` only carries `:payloads`/`:cwd`/`:worktree-root`/`:git-common-dir`/`:timeout` into `op!` (SPEC-004-D003.C1).
- **Hook gating reads op metadata; unknown ops fail before hooks.** `handle-invoke!` resolves the op first, so an unknown name returns the registry's available-names domain error *before* any `:payload/received` hook runs (MI3). Gating then keys on the entry's `:hook-class`: `:mutating` runs payload hooks (context `{:request/source :json-socket :request/operation :invoke :request/id … :request/args <decoded envelope> :request/options {} :op/name <canonical>}` plus the runner-added `:hook/type`), `:read` skips them, preserving the old read exemption (SPEC-004.C80 superseded).
- **Streaming = header/lines/terminator, run synchronously; `op!` accreted `:op/emit!`.** For `:stream? true` ops the socket writes `{"protocol_version":1,"request_id":…,"stream":true}`, then the handler's emitted values verbatim (one flushed line each), then `{"protocol_version":1,"request_id":…,"done":true,"success":true,"result":<return>}` or the `success:false`/`error:<envelope>` form. I used `"protocol_version"` (not the task's shorthand `protocol`) to stay consistent with the existing frame identity fields the Go relay (task 8) already parses. `op!` gained a `contains? envelope :emit!` → `:op/emit!` branch (weaver alpha); the socket injects the emit fn into the envelope. A hook rejection before the header yields a single error frame; a handler failure after the header yields an error terminator. Stream ops run on the connection thread (no future), so `emit!` and the terminator are single-writer.
- **Deadlines: single-result ops only; `:standard` gets a defined 10s server default.** No server-side deadline existed before this cutover, so `:standard → the existing default socket deadline` had no literal value to reuse; I defined `default-standard-deadline-ms` = 10000, mirroring the client's historical core-request protocol deadline (`cli/internal/client` `RequestDeadline`), and documented it in the socket ns. Effective deadline = envelope `timeout` (ms) if present, else `:standard`→10s / `:unbounded`→none. **Cancellation semantics (documented per MI4):** the op runs in a `future`; on expiry the connection returns `operation/deadline-exceeded` (domain) and `future-cancel`s with interruption, so the handler's blocking work aborts and the future's result is discarded — the connection writes exactly the timeout frame, no orphan success follows. Interruption is cooperative (a mutation already mid-commit is not rolled back). **Follow-up for task 9:** long-blocking ops that previously relied on the client long-deadline special case (agent `await`, `flow-await`, delegation) default to `:standard` today and must be reclassified `:deadline-class :unbounded` (or always invoked with an explicit `--timeout`) before the repo spool cutover routes them over the socket; the batteries CRUD ops are all fast and safe under 10s. **Stream deadline scope:** envelope `timeout` on a `:stream?` op is not enforced mid-stream (streams run unbounded on the connection thread); this is an accepted narrowing for the cutover, not a spec conflict, since C5 makes streams default to unlimited and MI4's orphan-write concern is about single-result mutating ops.
- **Signal shutdown = a JVM shutdown hook in `-main` (MI2).** `runtime/-main` installs a `Runtime/addShutdownHook` that runs the existing `stop!` clean path (transports down, storage closed, `weaver.edn`/`weaver.json`/`weaver.sock` removed) on SIGTERM/SIGINT (and normal exit) — the portable handler for both signals. Signal delivery is not unit-tested in-JVM (the allowed thin main-path hook); artifact cleanup on programmatic `stop!` is already covered by `runtime-stop-removes-metadata`.
