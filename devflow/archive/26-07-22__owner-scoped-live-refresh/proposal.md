## PROP-Olr-001.P1 Problem

Skein's runtime is a live image, but changing that image requires callers to orchestrate loader phases directly. The blessed path exposes approval synchronization, source reload, module activation, and global config replay as separate `sync!`, `reload-spool!`, `use!`, and `reload!` operations. A source edit therefore needs code reload followed by targeted activation or a global reload. A global reload clears unrelated registries, queued events, failure history, and module-use state before replaying every startup file.

This does not match Skein's primary operating model. Active DAGs of delegated agents must continue while trusted config changes beneath them. A workspace may replace a harness alias or routing policy so future runs use another provider while already-launched runs continue without interruption. Process replacement or global teardown for that change would destroy the capability the live runtime exists to provide.

Userland has already rebuilt partial reconciliation several times. The workspace `defop`, `defquery`, `defpattern`, and `defrule` helpers each keep a namespace-keyed shadow registry. File-backed modules conventionally clear their shadow contribution with a top-level `forget-*` call before remembering the declarations in current source, then copy those declarations into the live registry through `install-*`. Without the forget step, deleted source declarations remain in the shadow registry. Deletion from core destination registries still depends on global reload clearing them; spool-state registries survive that clear. The convention also differs between `:file` and `:ns` modules.

Loaded-code accounting has the same omission problem. Card `uxc5f` records that shrinking a loaded root's `deps.edn :paths` or deleting a loaded source file can remove that namespace from both current fingerprint inputs while its Vars and classes remain live. The runtime may then report a clean sync and clear pending-generation state even though loaded code has no current source owner.

The missing domain primitive is complete replacement of one owner's live contribution, coupled with cumulative accounting for code that has ever entered the image. Skein needs one deep refresh boundary that preserves live mutation while containing its blast radius and reporting what cannot be undone.

## PROP-Olr-001.P2 Goals

- **PROP-Olr-001.G1:** Preserve the Emacs-like live runtime as the default workflow. Config, source, harness, routing, workflow, and registration changes become live without interrupting unrelated work or replacing the weaver process when the JVM can continue safely.
- **PROP-Olr-001.G2:** Give callers one intent-level refresh operation that owns approval synchronization, source reload, affected-module reconciliation, dependency ordering, and a data-first result. Callers do not choose or order loader phases for the common path.
- **PROP-Olr-001.G3:** Partition replaceable declarative runtime entries by a stable module activation key. A successful refresh replaces that owner's complete core contribution, including deletions, while unrelated owners remain untouched. Defaults, workspace entries, and other legitimate contributors occupy distinct owner partitions. Domain registries provide their own complete-owner deletion contract through reconcile APIs. Live resource handles and in-flight tracking are carried or reconciled separately, never replaced as declaration data.
- **PROP-Olr-001.G4:** Make definition binding time contractual per definition family and execution unit. A running OS process or interactive session is never replaced by a harness edit; each later launch, retry, resume, workflow transition, review dispatch, scheduled delivery, event dispatch, hook invocation, or op invocation follows its own declared snapshot-or-live-resolution rule. Existing live re-pointing, such as devflow resolving the current constructor at a later stage transition, remains available where it is intentional.
- **PROP-Olr-001.G5:** Keep resource and effect semantics honest. Registration replacement may be atomic; arbitrary Clojure effects, durable writes, external calls, thread creation, and JVM class redefinition are not presented as transactional.
- **PROP-Olr-001.G6:** Track loaded source cumulatively by approved root and namespace. Source removal, path shrinkage, namespace rename, and other loaded-minus-current differences remain visible as residual code until a clean process cutover removes them. This goal fully subsumes card `uxc5f`.
- **PROP-Olr-001.G7:** Replace fragmented sync/use/pending introspection with one joined view of desired approvals, materialized roots, loaded-ever code, active owner contributions, retained resources, residual code, hard conflicts, and the most recent refresh outcome.
- **PROP-Olr-001.G8:** Retain trusted sharp tools for direct REPL experimentation and recovery, but keep their weaker guarantees explicit and outside the normal refresh contract.
- **PROP-Olr-001.G9:** Use TEN-000@1 to remove the current lifecycle rather than retain permanent aliases, deprecations, or migration shims. Preserve code-only and phase-level operations as explicitly advanced seams where they serve live-image diagnosis or recovery, not as decisions every caller must orchestrate. Update every affected first-party and currently sole-consumer peer spool in one coordinated consumer cutover.
- **PROP-Olr-001.G10:** Preserve asynchronous progress through refresh. Each dispatch snapshots the contribution it began with, later dispatches see the replacement, and domain engines retain their existing defer/retry behavior when a named dependency is temporarily unavailable. Refresh does not require globally stopping the event lane or active workers.

## PROP-Olr-001.P3 Non-goals

- **PROP-Olr-001.NG1:** No sandbox or rollback mechanism for arbitrary trusted Clojure. A source load or activation function may still produce effects before failing; the result must report that truthfully.
- **PROP-Olr-001.NG2:** No guarantee that JVM classes, deleted Vars, old instances, protocol implementations, multimethod state, or captured closures can be unloaded from a running process.
- **PROP-Olr-001.NG3:** No file watcher or automatic refresh daemon. Refresh remains an explicit trusted runtime or REPL action.
- **PROP-Olr-001.NG4:** No public JSON CLI configuration authoring surface. Rich config and refresh remain trusted Clojure workflows under TEN-006.
- **PROP-Olr-001.NG5:** No interruption or replacement of an already-running OS process or interactive session merely because its harness alias, reviewer roster, executor, or routing policy changes. Pending launches, retries, resumes, and later workflow transitions are separate binding points and may intentionally resolve updated definitions.
- **PROP-Olr-001.NG6:** No claim that the affected peer spools are unversioned. This workspace currently consumes `agent-harness.spool` v7, `kanban.spool` v4, and `devflow.spool` v2. Immutable old pins remain reproducible. By explicit owner direction, this feature may publish an in-place breaking next marker and waive those repositories' normal accretion/compat-alarm rule because this workspace is their only current consumer; the release record must name the exception and expected compatibility-alarm failure.

## PROP-Olr-001.P4 Proposed scope

- **PROP-Olr-001.S1:** Reshape the blessed runtime workspace API around owner-scoped module declaration, live refresh, and joined status. The current phase-level lifecycle functions cease to be peer operations on the normal public path; code-only reload and other load-bearing diagnostic seams remain available under explicitly advanced names.
- **PROP-Olr-001.S2:** Add stable module-owner identity and complete-contribution replacement to the declarative entries in the core registries: CLI operations, named queries, weave patterns, lifecycle hooks, and event handlers. Registry reads remain deterministic and data-first; unauthorized cross-owner collisions fail loudly before publication, while an explicit replacement names the displaced owner and key and remains visible in status. The owner survives namespace and source-file renames.
- **PROP-Olr-001.S3:** Provide owner-complete declaration replacement to spool-defined registries whose entries must reconcile live, including Chime rules, workflow constructors and executors, harnesses and aliases, reviewer rosters, benchmark harnesses/suites/extractors, kanban registrations, and workspace macro declarations. Each domain specifies how default and custom owner partitions compose. Domain registration effects such as Chime baseline seeding and cron wake arming remain in domain APIs rather than a generic map transaction.
- **PROP-Olr-001.S4:** Define module refresh outcomes for successful replacement, unchanged contribution, skipped optional module, validation refusal, partial source reload, effect failure, retained previous contribution, residual loaded code, and hard classpath conflict. Results name affected owners, roots, namespaces, registrations, and remedies.
- **PROP-Olr-001.S5:** Specify binding points for every affected named-definition family. At minimum this covers CLI op handlers, event handlers, lifecycle hooks, workflow routes and executor function values, harness aliases and backends, reviewer rosters, Chime rules, cron's job table plus scheduler wake, and kanban tracker functions. A durable scheduler wake captures its handler symbol and payload when scheduled, resolves the current Var at each delivery, and is not rewritten by owner replacement; removed or renamed handlers require explicit retained-wake reporting and migrate, cancel, or reschedule policy. Cron separately reconciles its runtime job declaration with the durable `cron/<id>` wake whose handler remains the stable cron dispatcher.
- **PROP-Olr-001.S6:** Replace current-source-only generation accounting with a cumulative loaded-root and loaded-namespace ledger. Namespace ownership is attributed at load time to the approved root and stable module that supplied the source; classpath-owned Skein API and batteries namespaces are represented separately. Joined status computes current provisions, cross-root transfers, and residuals against that ledger, never clears a residual on an unclassified or failed root, and blocks only changes that actually conflict with retained JVM state.
- **PROP-Olr-001.S7:** Replace workspace `remember-*`/`forget-*`/`install-*` reload workarounds with owner-aware declaration collection. Keep concise `defop`, `defquery`, `defpattern`, and `defrule` authoring forms where they still earn their place.
- **PROP-Olr-001.S8:** Preserve runtime-owned resource state and explicit lifecycle policy. Declarative entries are replaceable; live process/session handles, in-flight tracking, executors, timers, and durable coordination state are carried or reconciled through versioned spool-state and domain lifecycle functions. Resource-bearing spools declare whether refresh reuses, reconciles, replaces, or retains each state component; failures remain visible without globally clearing unrelated runtime state.
- **PROP-Olr-001.S9:** Coordinate the cutover across Skein; first-party `workflow` (including the shell executor), `roster`, `text-search`, `chime`, and `cron`; workspace macros/config; all three `agent-harness.spool` v7 roots (`agent-run`, `delegation`, `bench`); `kanban.spool` v4; and `devflow.spool` v2 where their declaration or binding contracts change. Update pins, docs, tests, generated API references, compatibility checks, and release markers together. Record the owner-approved in-place compatibility exception for each peer whose previous-marker alarm is expected to fail.
- **PROP-Olr-001.S10:** Preserve uninterrupted delegated work throughout design and implementation by leaving the canonical coordination world's pins unchanged. The final coordinated migration may require one explicitly approved, quiescent weaver generation replacement when base-classpath namespaces, loaded roots, or Maven coordinates change; it occurs only after active runs drain and every new pin is validated in disposable worlds. Ordinary refresh behavior after migration remains live and does not inherit that one-time cutover cost.

## PROP-Olr-001.P5 Open questions

None block spec planning. The spec-plan must choose the exact owner-contribution data shape, registry publication mechanism, module declaration form, residual-versus-hard-conflict classification, and peer-spool release-marker policy while satisfying the goals and scope above.
