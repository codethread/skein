# Spool `:needs` targeting skein-shipped namespaces

**Document ID:** `RFC-018`
**Status:** Open
**Date:** 2026-07-03
**Related:** [REPL API](../specs/repl-api.md) (SPEC-003.C24–C29, manifest grammar at line 99), [Weaver Runtime](../specs/daemon-runtime.md) (SPEC-004.C40/C50/C93/C94), [Registry-free git distribution for spools](../archive/26-07-03__spool-git-distribution/rfcs/2026-07-03-spool-git-distribution.md) (RFC-017), [Spool git distribution proposal](../archive/26-07-03__spool-git-distribution/proposal.md) (SGD-PROP-001.Q1), [Writing shared spools](../../docs/writing-shared-spools.md)

## RFC-018.P1 Problem

RFC-017 gave distributed spools a `spool.edn` manifest whose `:needs` names the coordinates a spool builds on. `:needs` keys are `spools.edn` **coordinate** symbols, verified against the approved, successfully-synced set (SPEC-003 manifest grammar; SPEC-004.C93). This only expresses dependencies on *other approvable spools*.

But shipped skein namespaces — the blessed `skein.api.*.alpha` API and the shipped reference spools such as `skein.spools.workflow` — come from the Skein checkout on the **weaver classpath** and deliberately do **not** require `spools.edn` approval (SPEC-003 line 85; SPEC-004.C40/C50). They have no approvable coordinate, so a distributed spool cannot name them in `:needs` at all.

The first concrete case is the extracted `devflow.spool`, which is built on `skein.spools.workflow`. Its manifest carries no `:needs` entry for that dependency because the grammar has no way to spell it. The real requirement is therefore invisible to `sync!`/`use!` and only surfaces late — at `require`/call time, cryptically — instead of loudly at activation. That is the exact fail-late gap TEN-003 exists to close.

This was deferred as SGD-PROP-001.Q1 ("Whether `:needs` may later name skein's own blessed namespaces … is deferred follow-up scope"). RFC-017.REC1 anticipated it: "the manifest grammar can accrete a level map later without breaking existing manifests if real demand appears." The demand has appeared.

## RFC-018.P2 Goals

- **RFC-018.G1:** A distributed spool can declare, in its `spool.edn`, that it builds on a skein-shipped namespace/spool.
- **RFC-018.G2:** `sync!`/`use!` verify that declaration against the runtime's own provided surface (the weaver classpath), loudly, reusing the existing unmet-needs reporting and `use!` gate.
- **RFC-018.G3:** Verification stays mechanically honest — presence, not a self-asserted compatibility claim — consistent with RFC-017's "ship shape-checking honest about being shape-checking" principle.
- **RFC-018.G4:** Only genuinely spool-facing skein tiers are declarable. The blessed `skein.api.*.alpha` API and shipped `skein.spools.*` reference spools are the entire declarable surface; `skein.core.*` (internal), `skein.repl` (human), and `skein.userland.alpha` (strict downstream) are non-declarable by contract and a need naming them is a loud authoring error, not a silently-checked classpath probe (AGENTS.md namespace tiers; SPEC-003.P5 tier map).
- **RFC-018.G5:** The `skein` coordinate group is reserved once, explicitly, across the whole spool-coordinate contract so the reserved-need form cannot be confused with a workspace-chosen approvable coordinate. Manifests that use no `skein/…` symbol validate exactly as before; a manifest or `spools.edn` that did use one fails loudly rather than being silently reinterpreted.

## RFC-018.P3 Non-goals

- **RFC-018.NG1:** No approval or fetch of shipped code. It already runs with the checkout's authority and is already present; consent stays per-coordinate for *approvable* code only (RFC-017.NG2). A shipped-namespace need never touches the network or the approved set.
- **RFC-018.NG2:** No new `spools.edn` coordinate *kind* and no new approval/fetch path. The approved-spool config grammar accretes exactly one constraint — the `skein` symbol-namespace is reserved and rejected as an approvable coordinate (RFC-018.REC1) — so the reserved-need form can never collide with a workspace-chosen coordinate. No `:local/root`/`:git/*` mechanics change.
- **RFC-018.NG3:** No version numbers, ranges, or integer accretion levels (RFC-017.O2 stays rejected; see O4 below).
- **RFC-018.NG4:** No resolver, no transitive shipped-dependency walking.

## RFC-018.P4 Options

| ID | Summary | Pros | Cons |
| --- | --- | --- | --- |
| RFC-018.O1 | Status quo: shipped dependencies stay undeclared; `:needs` names only approvable coordinates | Zero new surface | The manifest lies by omission; `devflow.spool`'s real dependency on `skein.spools.workflow` is invisible; unmet requirements surface at `require`/call time, not at `sync!`/`use!` — a TEN-003 fail-late gap |
| RFC-018.O2 | Overload existing `:needs` keys to accept a bare namespace symbol alongside coordinate symbols, discriminated heuristically | No new key | Coordinates and namespaces are both symbols; any heuristic (dotted vs. slashed, dictionary of known shipped ns) is brittle and silently mis-routes a mistyped coordinate into a classpath check or vice versa (TEN-003 violation via ambiguity) |
| RFC-018.O3 | Reserved `skein/…` pseudo-coordinate in `:needs` (e.g. `{skein/skein.spools.workflow nil}`), with the `skein` group **reserved across the coordinate contract** (§REC1), tier-restricted to the declarable skein tiers, and resolved against the runtime's **provided surface** by require-presence, never the approved set | Reuses the `:needs`/`:unmet-needs`/`use!`-gate machinery; a single reserved group makes reserved-vs-approvable unambiguous without a heuristic; check is mechanically verifiable (does the named namespace require cleanly in the weaver runtime?); no approval/fetch since the code is already trusted and present; existing manifests that use no `skein/…` key accrete without change | Introduces a reserved coordinate group authors must learn; requires the runtime to define what "provided surface" means; the reservation is a one-time loud breaking constraint on the previously-unreserved `skein/*` coordinate space (§REC1 migration) and needs a static tier check to keep non-spool-facing namespaces out (§REC1) |
| RFC-018.O4 | The runtime reports **integer accretion levels** for its shipped namespaces; `:needs` declares a minimum level (`{skein/skein.spools.workflow 3}`) | An in-band "how far has this API accreted" signal; levels would be skein-asserted (the maintainer that actually owns `skein.api.*.alpha` accretion), not spool-asserted as in RFC-017.O2 | Rejected as RFC-017.O2 already was: a compatibility number nobody mechanically verifies against behavior is fail-quietly trust-bait (TEN-003). The require-presence check (O3) already catches the honest, verifiable failure (missing namespace/var); drift *within* a present namespace is exactly the class a hand-maintained integer claims to catch and exactly the class that gets forgotten. The only new fact vs. O2 — that skein asserts its own levels — does not change that the number stays unverified. Not relitigated further absent new evidence |
| RFC-018.O5 | Collision-proof by construction: a **separate manifest key** (`:runtime-needs`, a vector/set of bare skein namespace symbols) instead of overloading `:needs`, so no reserved coordinate group is needed and no existing `:needs` coordinate can collide | Cannot collide with any workspace-chosen coordinate symbol, so no reservation and no migration break; keeps approvable-coordinate `:needs` and runtime `:runtime-needs` visibly distinct | Splits the one-map/one-`:unmet-needs`-list model into two parallel surfaces that `sync!` reporting and the `use!` gate must both walk; the same static tier restriction (declarable tiers only) is still required, so it removes the reservation cost but not the tier check; two keys is more grammar and doc surface for authors than one accreted group. A viable fallback if the `skein/*` reservation is judged too invasive |

## RFC-018.P5 Recommendation

- **RFC-018.REC1:** Adopt **Option O3**. `:needs` accretes a reserved coordinate group: a key whose symbol namespace is `skein` (e.g. `skein/skein.spools.workflow`) denotes "provided by the runtime; verify by require-presence of the named namespace." Concretely:
  - **Reserving the `skein` group (SPEC-003.P5 coordinate accretion):** the `skein` symbol-namespace is reserved across the whole spool-coordinate contract, not just inside `:needs`. Today coordinate symbols are workspace-chosen and unreserved (SPEC-003.P5; `docs/writing-shared-spools.md` "the approving workspace chooses the coordinate symbol"), so `skein/foo` could in principle be an ordinary approvable coordinate; this RFC closes that hole. After acceptance: (a) a `spools.edn`/`spools.local.edn` coordinate whose symbol namespace is `skein` is a **loud structural config error** (`:reserved-coordinate`), never an approvable entry — the same fail-loud class as an unknown top-level key; and (b) `skein/…` inside `:needs` is *only* the reserved runtime-need form below, never a reference to an approved coordinate. Because approvable and reserved coordinates now live in disjoint symbol-namespaces, there is no heuristic and no ambiguity (contrast O2).
  - **Migration / compat:** the reservation is a deliberate one-time break scoped to the (undocumented, never-recommended) `skein/*` coordinate namespace. A workspace that had approved a `skein/*` coordinate gets a loud `:reserved-coordinate` config error naming the offending symbol, with the migration being a one-line rename to any non-`skein` group; nothing is silently reinterpreted. Every manifest and config that uses no `skein/…` symbol is unaffected (RFC-018.G5).
  - **Tier restriction (SPEC-003.P5 tier map):** a reserved need may name **only a declarable tier** — the blessed `skein.api.*.alpha` API or a shipped `skein.spools.*` reference spool. A `skein/…` need naming any other skein namespace — `skein.core.*` (internal), `skein.repl` (human surface), `skein.userland.alpha` (strict downstream), or a non-`skein.*` name — is a static authoring-contract violation and fails that spool's sync loudly as `:manifest-invalid` carrying `:reason :non-declarable-tier` and the offending namespace, without loading the root. This is decided syntactically from the namespace name, before any classpath probe, so a non-spool-facing namespace can never be back-doored into a declarable dependency even if it happens to require cleanly.
  - **Grammar (SPEC-003.P5 accretion):** `:needs` keys may be an approvable coordinate symbol (unchanged) **or** a reserved `skein/<namespace>` symbol whose bare name is a declarable-tier skein namespace. Values keep the existing shape but a `skein/…` need takes only `nil` (a `:suggest {:git/url …}` hint is meaningless for shipped code and is a loud `:manifest-invalid` on a reserved need). Existing manifests, which use no `skein/…` keys, validate exactly as before.
  - **Provided surface:** the runtime's provided surface is the set of declarable-tier skein namespaces loadable from the weaver classpath for the selected workspace — the blessed `skein.api.*.alpha` API and the shipped `skein.spools.*` reference spools (SPEC-003 line 85; SPEC-004.C40/C50). Verification is a require-presence check ("does this namespace require cleanly in the weaver runtime?"), never a level comparison. The tier restriction above bounds *what may be asked*; the provided surface bounds *what is present*.
  - **`sync!` (SPEC-004.C93 accretion):** a tier-valid reserved need that resolves in the provided surface is satisfied and touches no network and no approved set. A tier-valid but unresolvable one is recorded on the needing spool's still-`:loaded` result as an `:unmet-needs` entry with a **new reason `:not-provided-by-runtime`**, distinct from the existing `:not-approved`/`:sync-failed`. (A tier-invalid reserved need never reaches this stage — it already failed the spool's sync as `:manifest-invalid`.) Needs never trigger fetching (RFC-017.NG2 preserved).
  - **`use!` (SPEC-004.C94 unchanged in shape):** the existing activation gate already refuses on any `:unmet-needs`; a `:not-provided-by-runtime` entry gates identically, throwing under `:required? true`. No new check order.
- **RFC-018.REC2:** This keeps every manifest claim mechanically verifiable — the same honesty bar RFC-017 set for the SHA pin and `:provides` — and reuses one `:needs` map and one `:unmet-needs` list rather than adding a parallel key. The one-map model is only defensible because the reserved group is disjoint from approvable coordinates and tier-restricted; O5's separate `:runtime-needs` key is the fallback if reserving `skein/*` across the coordinate contract is judged too invasive, accepting two parallel need surfaces in exchange for needing no reservation.

## RFC-018.P6 Consequences

- **RFC-018.C1:** SPEC-003 (REPL API) accretes: (a) the `skein` symbol-namespace reserved across the coordinate grammar, with a `skein/*` `spools.edn`/`spools.local.edn` coordinate becoming a loud `:reserved-coordinate` structural config error; and (b) the reserved `skein/…` `:needs` form with its `nil`-only value constraint and declarable-tier restriction (`skein.api.*.alpha` / `skein.spools.*` only), where a non-declarable-tier target fails the spool's sync as `:manifest-invalid` (`:reason :non-declarable-tier`). SPEC-004 (Weaver Runtime) `sync!` gains the provided-surface resolution and the `:not-provided-by-runtime` unmet-needs reason; `use!` gating is unchanged in shape.
- **RFC-018.C1b:** The `skein/*` coordinate reservation is a one-time, loud, deliberate break of the previously-unreserved coordinate space (RFC-018.G5). It affects only workspaces/manifests that had adopted a `skein/*` coordinate — an undocumented, never-recommended shape — and its migration is a rename to any non-`skein` group. `docs/writing-shared-spools.md` states the reservation alongside its coordinate-symbol guidance.
- **RFC-018.C2:** `devflow.spool` can finally declare its `skein.spools.workflow` dependency, closing SGD-PROP-001.Q1. Other extracted/distributed spools that build on shipped spools or the blessed API gain the same expressiveness.
- **RFC-018.C3:** No new consent path, coordinate kind, fetch behavior, or CLI surface; the change lives entirely in manifest validation and sync/gate reporting (TEN-004, TEN-006 preserved).
- **RFC-018.C4:** `docs/writing-shared-spools.md` grows guidance on declaring shipped-namespace needs and reading the `:not-provided-by-runtime` outcome.
- **RFC-018.C5:** Whether the runtime should *enumerate* a discoverable provided-surface list (for tooling and `:suggest`-style ergonomics) is a smaller optional follow-up, not required by this RFC.

## RFC-018.P7 Outcome

- **RFC-018.OUT1:** Open — awaiting a decision from the repository owner. On acceptance, implementation stages the SPEC-003/SPEC-004 accretions and updates `devflow.spool`'s manifest to declare its `skein.spools.workflow` need.
