# Weaver Runtime

**Document ID:** `SPEC-004` **Status:** Implemented **Last Updated:** 2026-07-09 **Related RFCs:** [RFC-002 Task Query DSL](../rfcs/2026-06-24-task-query-dsl.md), [RFC-003 Fast JSON Socket CLI](../archive/26-06-25__go-cli-migration/rfcs/2026-06-25-fast-json-socket-cli.md), [RFC-004 Go CLI Migration](../archive/26-06-25__go-cli-migration/rfcs/2026-06-25-go-cli-migration.md) **Code:** `src/skein/core/weaver`, `src/skein/core/client.clj`, `src/skein/api/weaver/alpha.clj`, `cli/`

## SPEC-004.P1 Purpose

The weaver runtime is the long-lived local Clojure process that owns strand storage access, query execution, and runtime extension state for CLI and REPL clients.

## SPEC-004.P2 Runtime model

- **SPEC-004.C0a:** The alpha mill router runtime adds a local Go `mill` process as the known XDG entrypoint for routing/supervision. The mill owns process routing metadata only; the weaver remains the semantic runtime that owns storage, query execution, runtime registries, trusted config evaluation, hooks/events, and REPL semantics.
- **SPEC-004.C1:** A weaver owns exactly one active SQLite datasource, one in-memory named-query registry, one in-memory read-only view registry, one in-memory weave-pattern registry, one in-memory CLI operation registry, one in-memory lifecycle hook registry, one in-memory event handler registry with asynchronous dispatch state, one in-memory approved-spool sync state, one in-memory module-use registry, one in-memory spool-state registry, one runtime-owned clock component (SPEC-004.C1a), and one runtime-owned scheduler subsystem state (held through the spool-state registry per SPEC-004.P10d) for its lifetime.
- **SPEC-004.C1a:** A weaver runtime owns exactly one clock component for its lifetime: a runtime-map slot holding a zero-arg function that returns the current `java.time.Instant`, defaulting to the real wall clock (`#(Instant/now)`). Core timer-driven subsystems read the current instant through this clock component via a `skein.core.weaver.runtime` core-tier accessor, never by calling `(Instant/now)` directly for due-detection; cosmetic failure/running-marker timestamps are not due-detection and read the wall clock freely. The one blessed, data-first read is `skein.api.runtime.alpha/now` (SPEC-003.C17), which returns the current `Instant` — not the clock function — for the runtime passed as its first argument; the clock component and its clock-pump registry stay off the alpha surface (SPEC-005.C5a). Installing a manual clock and advancing it are author-side test controls, `skein.test.alpha/set-clock!` and `skein.test.alpha/advance!` (SPEC-003.C28a): one `advance!` moves the runtime clock forward by a strictly positive `java.time.Duration` — a non-positive or backwards duration fails loudly (TEN-003) — then runs every registered clock-consumer pump synchronously before returning, so a subsystem that arms a real `ScheduledExecutorService` timer off the clock (the scheduler; the userland cron spool) releases its now-due work deterministically under a manual clock rather than on wall delay. Default (production) behaviour is unchanged: the default clock returns real wall-clock instants and real executors fire on real time.
- **SPEC-004.C2:** A weaver exposes two local transports: nREPL for Clojure REPL/client workflows and a JSON Unix domain socket for the public Go CLI.
- **SPEC-004.C3:** Transports are local-only by default: nREPL binds to loopback, and the JSON CLI transport uses a Unix domain socket under the selected runtime state directory.
- **SPEC-004.C4:** A weaver workspace is selected by workspace. The CLI supplies this selected workspace after explicit `--workspace` handling or repository-canonical `.skein` discovery; Clojure entry points fail loudly when no selected workspace is provided. Without `--workspace`, linked worktrees for one supported non-bare Git repository converge on the same canonical repository `.skein` selected workspace.
- **SPEC-004.C5:** Runtime state for a selected workspace lives in the explicit selected state dir supplied at weaver startup, and weaver-owned data lives in the explicit selected data dir supplied at weaver startup. Mill derives those dirs from the canonical selected config identity, so linked worktrees that share the default workspace also share runtime/data/metadata identity.
- **SPEC-004.C6:** A selected workspace is a trusted config workspace only; config files load from that directory while runtime state and data may live in independent XDG dirs selected by mill. Mill owns Skein source checkout resolution for launching the weaver and any thin nREPL attach client used by `weaver repl`; the weaver receives explicit `--workspace`, `--state-dir`, `--data-dir`, and optional `--name` values and does not read source from `config.json`.
- **SPEC-004.C7:** The weaver owns storage selection. By default it uses `skein.sqlite` under the selected data dir. Public clients do not choose storage paths.
- **SPEC-004.C7a:** Weaver-owned SQLite access is configured for concurrent in-process writers from sockets, event handlers, and spool worker threads to wait briefly on write-lock contention rather than fail immediately. Mutating transactions acquire write intent at transaction start so lock waiting happens at a predictable boundary.
- **SPEC-004.C8:** A selected workspace may have at most one running weaver. Starting another weaver for the same selected workspace fails loudly unless stale socket/metadata can be proven dead and cleaned.
- **SPEC-004.C8a:** In one JVM, runtime existence is distinct from ambient publication. `skein.core.weaver.runtime/start!` publishes the new runtime as the process ambient runtime by default and fails loudly if another runtime is already published. With `:publish? false`, it starts an unpublished runtime; any number of unpublished runtimes may coexist when they use independent selected workspaces. `stop!` clears ambient publication only when the stopped runtime owns it.
- **SPEC-004.C8b:** `skein.api.current.alpha/runtime` resolves the current runtime in this order: thread-local runtime binding, published ambient runtime, then a loud failure. Startup and reload config evaluation and built-in op registration bind the serving runtime so trusted config code using `current/runtime` works for unpublished runtimes.
- **SPEC-004.C9:** Multi-weaver use is explicit workspace selection: different selected workspaces create separate weaver/socket/metadata/data/init workspaces. Explicit `--workspace` is the intended isolation mechanism for Skein development, tests, and worktree-local experiments. Public request payloads do not carry cwd, repo metadata, storage paths, or source checkout paths as workspace identity.

## SPEC-004.P3 Runtime metadata and discovery

- **SPEC-004.C9a:** The mill router publishes its entrypoint under Skein's XDG state root: `$XDG_STATE_HOME/skein` or `~/.local/state/skein` when `XDG_STATE_HOME` is unset. Mill metadata lives at `mill.json`, and the mill JSON socket lives at `mill.sock`.
- **SPEC-004.C9b:** Mill derives per-workspace runtime directories under `<state-root>/weavers/<hash>`, where `<hash>` is a filesystem-safe hash of the canonical selected config identity.
- **SPEC-004.C9c:** The mill JSON socket protocol itself — frame shapes, operation set, `mill/*` error codes, protocol version, and the `mill.json` field set — is internal transport glue between the shipped Go binaries and carries no contract (SPEC-005.C6). Clients consume mill through the CLI command surface (SPEC-002), never by dialing `mill.sock` directly.
- **SPEC-004.C10:** The JSON socket path is fixed within the selected state workspace as `weaver.sock`.
- **SPEC-004.C11:** Runtime metadata is published beside the fixed socket as `weaver.json` for Go clients and `weaver.edn` for Clojure clients.
- **SPEC-004.C12:** Runtime metadata records enough identity for clients to connect and verify intent: pid, weaver id, protocol version, friendly name, selected workspace, selected state dir, selected data dir, storage kind and label plus weaver-owned database path (file storage only) for status/debugging, nREPL endpoint data, JSON socket path, and startup time. The friendly name defaults to the selected workspace basename unless launch supplied an explicit name or the selected workspace config declared one (SPEC-002.C2/C2a). Metadata does not record source as selected workspace identity; source is mill-owned launch context.
- **SPEC-004.C13:** Public Go CLI clients dial the local mill (SPEC-002.C1, SPEC-004.C9a); mill discovers the selected-workspace weaver by state workspace and forwards the request over the weaver socket, verifying weaver identity/protocol on that leg. Trusted Clojure clients discover the weaver by selected workspace/state workspace and verify weaver identity/protocol over the socket directly. Neither needs a database path to locate metadata, and neither sends or compares database path as a request identity field.
- **SPEC-004.C14:** Stale, missing, mismatched, malformed, or unreachable runtime metadata is an error; clients must fail loudly rather than silently opening SQLite directly. Clients discover only `weaver.*` artifacts and do not fall back to old `daemon.*` metadata. `mill weaver list` discovers supervised children and `weaver.json` files under the mill state root, returning running/stale rows for known selected workspaces and failing loudly on malformed metadata.

## SPEC-004.P3a Weaver storage model

- **SPEC-004.C91:** A weaver runtime owns exactly one storage handle for its lifetime. The handle carries the storage kind, a diagnostic label, an optional canonical filesystem path, a next.jdbc-compatible connectable, and optional close behavior. Startup normalizes storage into the handle before database initialization and metadata publication, then opens SQLite with WAL journaling, a non-zero memory map size, and an enlarged page cache for every world. Runtime stop closes runtime-owned spool-state resources with `:close-fn` after transports and event dispatch stop but before storage closes, then closes weaver-owned storage resources. Failed startup follows the same ordering for any spool state initialized before the failure: close runtime-owned spool-state resources before storage, suppressing close-hook failures onto the original startup exception so cleanup cannot mask the startup cause, then close storage on the error path.
- **SPEC-004.C91a:** Weaver storage initialization creates dedicated scheduler tables (SPEC-004.P10d) beside the strand graph. Scheduler rows are weaver runtime coordination state owned by the scheduler contract, not user graph rows; they never appear in strand `list`/`ready`/query/traversal/burn/update paths.
- **SPEC-004.C91b:** Storage initialization validates any existing database against the current core schema and fails loudly on an incompatible one; there is no implicit migration (TEN-000: use a new database or migrate explicitly). It creates current-schema tables only after this compatibility check, so opening a legacy document-column database cannot leave a mixed partial schema behind.
- **SPEC-004.C92:** `:sqlite-file` is the default storage kind for normal weaver workspaces. It uses `data/skein.sqlite` under the world's weaver state directory — the `data_dir`/`database_path` reported by `mill weaver status --workspace <w>`, resolvable before any weaver starts — unless trusted runtime construction supplies another database file. The selected workspace directory holds the world's config, not its database.
- **SPEC-004.C93:** `:sqlite-memory` is available only to trusted test/runtime construction. It uses real Xerial SQLite JDBC with a weaver-owned held `java.sql.Connection` running the same schema and SQL code as file storage; data is weaver-lifetime only, closing the connection destroys the database, and later use of closed storage fails loudly. One held connection serializes writes, so memory storage is test storage, not production-like pooled storage. The public CLI cannot select storage kind.
- **SPEC-004.C94:** Metadata reports storage identity explicitly. EDN metadata uses `:storage-kind`, `:storage-label`, and `:canonical-db-path`; JSON metadata and socket status use `database_kind`, `database_label`, and `database_path`. For file storage the label and path are both the canonical database path; for memory storage the label is a stable weaver-lifetime diagnostic string and the path is explicitly nil/null — never a fake filesystem path. Kind/label/path consistency is validated fail-loud both when metadata is published and when clients read it.
- **SPEC-004.C95:** `skein.api.runtime.alpha/spool-state` is the sole accessor for runtime-owned per-key spool state, initialized once with `init-fn` and reused for the runtime lifetime. Spool-state entries deliberately survive `reload!` (unlike the registries C46 clears), so a spool whose state shape changed between deploys must not silently reuse a preserved value missing the new keys. The four-arg arity takes opts `{:version v :migrate-fn f}`: the declared `version` is stored as value metadata, and when a preserved value's stored version does not equal the currently declared `version`, the runtime deliberately reinits rather than reusing shape-mismatched state — running `init-fn` fresh after best-effort `:close-fn` release of the old value (preserving nothing), or, when `:migrate-fn` is supplied, handing the old value to `f` (which then owns the old value's resources) to produce the new one. Versioned state must support metadata. The three-arg arity is unversioned (version nil) and always reuses an existing value; consumers that store executors, schedulers, or other shape-sensitive resources declare a version and bump it whenever `init-fn`'s key set changes. The batteries read cap is runtime state set by trusted config through `skein.spools.batteries/set-read-limit!`; invalid cap values fail loudly. The opts map is validated at the call site — closed to `:version`/`:migrate-fn`, `:version` must be a non-nil integer/keyword/string, and `:migrate-fn` must be a function that requires a `:version` — so a misspelled key (`{:versoin 2}`) or malformed value fails loudly instead of silently degrading to the unversioned reuse path. When a preserved value mismatches the declared version, the reinit transition is serialized per runtime so `migrate-fn`/`init-fn` and the resources they build run at most once: concurrent accessors racing the same mismatch never each construct a replacement (a discarded loser would leak the executors/schedulers it built for the JVM lifetime), and a best-effort `:close-fn` failure during the release is warned rather than silently swallowed.
- **SPEC-004.C96:** `reload!` never leaves a world with no ops. The reload path clears registries and reinstalls built-in ops before re-running startup files (C46, C63c, C70); a startup file that throws midway leaves the built-in ops plus every userland registration that loaded before the failure in place, then rethrows loudly. The failure path does not re-clear registries — that would take already-registered userland ops down with the failure. Consumers that hold set-once registrations in reload-surviving spool-state (for example a worker-contract preamble extension) tolerate reload re-registration, distinguishing replay from conflict: an identical value is a silent no-op, while a different value (a genuine cross-spool clash) replaces the prior one rather than aborting startup — so a value change between deploys cannot leave the world unusable — but the clash is recorded as a durable, queryable signal (not stderr-only), so the conflict stays visible to operators and attention detectors after the warning has scrolled off.

## SPEC-004.P4 API boundary

- **SPEC-004.C15:** The `skein.api.*.alpha` namespaces are the semantic boundary used by trusted in-process code and fixed-form clients; their public functions take the target runtime as their first argument. Transport-specific eval strings, JSON envelopes, nREPL messages, and wire details are not the durable product API.
- **SPEC-004.C16:** Weaver API operations cover the current stripped strand surface and trusted runtime state helpers, each owned by its domain namespace. `skein.api.weaver.alpha` owns the primitives no domain namespace does: initialize storage as a trusted idempotent helper, add strand, update strand, supersede strand, show strand, list strands, ready strands, archive/unarchive one strand's attributes, declare/list durable acyclic relations, execute list/ready through a named query, and register/replace/list/resolve/invoke weaver-memory CLI operations. `skein.api.graph.alpha` owns the named-query registry (register one named query, load named queries, list registered query definitions, resolve a named query, list registered query metadata, explain a registered query) and the set-oriented primitives (`query-ids`, `burn-by-id!`, `burn-by-ids!`, `strands-by-ids`, `ancestor-root-ids`, `subgraph`, `incoming-edges`, and `outgoing-edges`). `skein.api.views.alpha` owns register/list/invoke weaver-memory views; `skein.api.patterns.alpha` owns register/list/resolve/explain/invoke weaver-memory weave patterns; `skein.api.events.alpha` owns register/unregister/list/inspect trusted event handlers and queue submission; `skein.api.hooks.alpha` owns register/unregister/list trusted lifecycle hooks; `skein.api.batch.alpha` owns transactional batch graph mutation payloads; `skein.api.runtime.alpha` owns reading approved spool config, syncing approved spools, inspecting approved-spool sync state, reloading selected workspace startup files, activating one module with `use!`, and inspecting module-use state.

- **SPEC-004.C16a:** Attribute archive operations are trusted Clojure config/REPL surface only. `archive!` marks all attributes, or an explicit non-empty key set, archived for one strand. `unarchive!` reverses that state. Both validate the strand id and requested keys before writing, commit atomically, and return a map carrying `:strand-id`, `:keys`, `:archived?`, and `:changed`. Archived keys remain visible to full point reads but are omitted from hot-tier reads: trusted in-process `list`, `ready`, and query execution, plus the CLI/agent lean projections layered on those reads. A later write to an archived key records fresh hot data for that key; untouched archived keys stay archived.

- **SPEC-004.C16b:** Worlds predating the EAV row schema have no in-tree migration path; reinitialize the world, or check out merge `c77332a` (PR #6), whose tree carries the verified migrate op.

- **SPEC-004.C17:** Weaver API return values are Clojure data with JSON-bearing database columns normalized before transport-specific formatting. Strand rows use `state`; they do not expose removed lifecycle fields such as `active`, `inactive_at`, `status`, or `final_at`. Trusted in-process point reads (`show`, `strands-by-ids`, and `subgraph`) decode attributes to full-fidelity Clojure maps including archived values. Trusted in-process `list`, `ready`, and query execution are hot-tier reads: they decode full values for non-archived attributes only and are unbounded by default for REPL/config workflows. The lean tier is layered above hot-tier reads at the CLI/agent op boundary, and the batteries `list`/`ready` command surface may call the trusted lean projections with an explicit result cap. That cap is a CLI/wire guard: storage probes `limit + 1` rows before attribute assembly, reports the total on overflow, and never truncates successful results.

- **SPEC-004.C17a:** The lean-tier omission descriptor is the shared `::specs/omitted-attribute-descriptor` contract. The canonical trusted spool reader `attr-get` fails loudly when handed a value conforming to that spec where a raw value is expected; its `ex-info` ex-data is `{:key <attribute-key> :strand-id <strand-id> :recovery "show <strand-id>"}` so a misrouted lean-projected strand surfaces immediately with enough context to recover.
- **SPEC-004.C18:** Weaver API failures preserve domain error information well enough for clients to exit non-zero with useful messages.

## SPEC-004.P5 nREPL transport

- **SPEC-004.C19:** nREPL remains available for direct live weaver REPL sessions, Clojure/JVM process clients, editor tooling, and fixed-form API helpers.
- **SPEC-004.C20:** Fixed-form Clojure clients call only weaver API forms generated by trusted client code; user CLI input for public JSON operations is passed as data arguments, not interpolated as executable code. Direct live REPL sessions are trusted and may evaluate arbitrary forms inside the weaver JVM.
- **SPEC-004.C21:** nREPL is not the public fast CLI transport; it remains the rich Clojure-native surface for direct live weaver REPL sessions, REPL helpers, and fixed-form Clojure client/API calls. `skein.repl` helpers evaluated inside the weaver JVM may use the active runtime for human ergonomics; explicit client/test contexts route fixed API forms through connected client metadata. Fixed-form client eval resolves the runtime by the connected server's nREPL port rather than by the process ambient runtime, so client calls against unpublished runtimes in a multi-runtime JVM route to the runtime that owns that nREPL server. The blessed `skein.api.*.alpha` domain namespaces take explicit runtime arguments and do not perform connected-client routing.

## SPEC-004.P6 JSON Unix socket transport

- **SPEC-004.C22:** The weaver exposes a local JSON Unix domain socket (`weaver.sock`) advertised in Go-readable runtime metadata (`weaver.json`). The public Go CLI does not dial it directly: CLI clients reach the local mill JSON socket (SPEC-004.C9a), and mill forwards the `invoke`/`status` frames to the selected weaver's socket (SPEC-002.C1). Trusted Clojure clients and peers dial the weaver socket directly (SPEC-004.C85/C88).
- **SPEC-004.C23:** JSON socket requests are one request per connection and include protocol version, request id, weaver id, operation name, operation arguments, and an empty options object reserved for future protocol use.
- **SPEC-004.C23a:** The JSON socket accept loop may serve multiple client connections concurrently. A long-running trusted operation on one connection must not starve unrelated status, stop, or graph requests on other connections.
- **SPEC-004.C24:** JSON socket responses include protocol version, request id, success flag, result on success, and a structured error envelope on failure. Error types distinguish domain, protocol, and transport failures. The type taxonomy and envelope shape are contract; concrete error `code` strings are contract only where a clause specifies them individually (`operation/deadline-exceeded` C26b, `query/not-found` C36b, `hook/failed` C84) — other code strings and human-readable message text may change (SPEC-005.C7).
- **SPEC-004.C25:** The JSON transport dispatches to weaver semantic operations rather than duplicating SQL or query logic in transport handlers.
- **SPEC-004.C26:** The socket operation set is `invoke` plus a minimal `status` health/identity operation. `invoke` carries the envelope `{name, argv, payloads, cwd, worktree_root, git_common_dir, workspace, timeout?, client{pid,version}}` as the operation arguments inside the existing request frame and dispatches to the op registry; envelope shape is validated (loudly) but argument validation belongs to ops via the blessed parser. Client identity is socket-level diagnostic data: visible to payload hooks in `:request/args`, not threaded into op handler context. Storage initialization is a weaver startup responsibility, not a public JSON socket operation.
- **SPEC-004.C26a:** There is no socket `stop` operation. Weaver shutdown is signal-driven: SIGTERM/SIGINT run the clean stop path (transports down, storage closed, `weaver.edn`/`weaver.json`/`weaver.sock` removed), and mill supervision owns sending the signal.
- **SPEC-004.C26b:** Request deadlines apply to single-result responses: the op's registered deadline class supplies the default (`:standard` = 10s, `:unbounded` = none); an envelope `timeout` (milliseconds) overrides it. On expiry the connection returns an `operation/deadline-exceeded` domain error and the handler future is cancelled with interruption — no orphan success frame follows a reported timeout (interruption is cooperative; a mutation already mid-commit is not rolled back). Stream-class ops run unbounded and are never server-deadline-bounded, including when an envelope timeout is present; cancellation is client disconnect, which aborts the emitting handler on its next write.
- **SPEC-004.C26c:** Responses self-describe as single or stream. A single result is the existing one-frame response. A stream-class op's response is a header frame (`stream: true` with frame identity), the handler's emitted values verbatim (one flushed NDJSON line each, via the `:op/emit!` context callback), and a terminator frame (`done: true` with success/result or error envelope). A hook rejection before the header yields a single error frame; a handler failure after the header yields an error terminator. One request per connection is retained; a streaming response holds its connection open without starving others (SPEC-004.C23a).
- **SPEC-004.C27:** Query registry mutation/authoring, pattern registry mutation, view registry operations, CLI operation registry mutation, event handler operations, and lifecycle hook operations have no socket operations; those remain REPL/trusted config workflows. Read-only registry introspection reaches the socket only through registered read ops (core `help`; batteries `query`/`pattern`).
- **SPEC-004.C28:** `status` is a minimal health/identity check: it validates the matched weaver over the socket and reports identity/health data for client verification and `mill weaver status` composition. Full status reporting is mill's concern (SPEC-002.C20); shutdown is signal-driven (SPEC-004.C26a).

## SPEC-004.P7 Configuration and user code

- **SPEC-004.C29:** Core weaver startup works without a user init file.
- **SPEC-004.C30:** The weaver loads selected workspace startup files in order: `init.clj`, then `init.local.clj`. Missing startup files are skipped. Load errors in present files fail startup loudly with file context and publish no ready metadata.
- **SPEC-004.C31:** Trusted init code runs after weaver memory/socket state exists and before runtime metadata is published, so it may initialize weaver-owned runtime state through startup helpers such as `skein.api.graph.alpha/register-query!` after capturing the runtime explicitly (for example via `skein.api.current.alpha/runtime`).
- **SPEC-004.C32:** Sandboxing, SCI execution, untrusted plugin isolation, remote weaver access, authentication, multi-user authorization, and daemon startup hooks for storage customization are outside the current runtime contract.

## SPEC-004.P8 Named query registry

- **SPEC-004.C33:** Named queries are data-first EDN DSL definitions stored in weaver memory, not executable user functions.
- **SPEC-004.C34:** Registry names are simple unqualified names; symbol and keyword forms of the same name resolve to one entry.
- **SPEC-004.C35:** Registry contents are not durable across weaver restarts. Users reload runtime query definitions through trusted startup config or REPL helpers.
- **SPEC-004.C36:** Missing named-query resolution fails loudly with a clear domain error.
- **SPEC-004.C36a:** Query metadata listing returns a vector of serializable entries ordered by canonical name. Each entry contains the canonical `:name`, declared `:params` (the allowlist from map query definitions; empty for vector definitions), and `:referenced-params` (parameter references discovered by walking the effective `:where` expression for `[:param kw]` forms, including relation positions in `[:edge/out ...]` / `[:edge/in ...]` predicates). Entries do not embed full definitions.
- **SPEC-004.C36b:** Query explanation returns one query's serializable caller guidance: the listing fields plus `:where` (effective where expression as a JSON-safe projection), `:definition` (normalized registered definition as a JSON-safe projection), `:where-form` / `:definition-form` (exact `pr-str` EDN strings for trusted users and docs), and a short `:summary` stating that invocation happens through `list --query` / `ready --query` with runtime params. Unknown names fail loudly with the existing `query/not-found` domain error including available names.
- **SPEC-004.C36c:** Query introspection reaches the socket only through the batteries `query` op (`strand query list|explain <name>`), which dispatches to the weaver API introspection helpers with the same trim/leading-colon name normalization as named-query application; blank names fail as domain errors at normalization.

## SPEC-004.P9 Runtime spool workspace model

- **SPEC-004.C37:** Runtime extensions are normal trusted Clojure spools/modules made available to the weaver through selected workspace startup (`init.clj` then `init.local.clj`) or live weaver REPL workflows.
- **SPEC-004.C38:** Extension code runs with weaver process authority. Sandboxing, untrusted execution, remote authorization, and capability restriction are outside this contract.
- **SPEC-004.C39:** Skein ships the blessed source-visible `skein.api.runtime.alpha` privileged runtime loader/config helper for spool-workspace workflows. Privileged loader/config helpers do not live under `skein.spools.*`; that namespace family is reserved for authorable spools and examples.
- **SPEC-004.C40:** Blessed `skein.api.*.alpha` namespaces and shipped reference spools are documented, tested, and used by examples. They are recommended maintenance paths, not enforcement boundaries; trusted code may require lower-level namespaces or use raw SQLite schema when it accepts compatibility cost.
- **SPEC-004.C41:** The selected workspace is a trusted alpha spool workspace root. User-owned config may include `config.json` with only `"configFormat":"alpha"`, shared `init.clj`/`spools.edn`, local `init.local.clj`/`spools.local.edn`, local source directories, and user code. The CLI bootstrap path may initialize missing workspace files/directories, but must not initialize a Git repository, persist source checkout paths, or overwrite existing user files.
- **SPEC-004.C42:** `spools.edn` and `spools.local.edn`, when present, declare approved weaver-wide spool coordinates of exactly one kind per entry. Local kind: `:local/root`, where relative entries resolve against selected workspace, absolute roots are accepted as explicit user-approved paths, and leading `~` and `~/` expand to the user home directory. Git kind: required `:git/url` (passed to system git verbatim; the user chooses the transport) and `:git/sha` (exactly 40 lowercase hex characters — the exact-content consent pin), optional `:git/tag` (human-readable label verified against the sha at fetch time), and optional `:deps/root` (relative subpath into the checkout, no leading `/`, `~`, or `..` segments; git-only). Mixed kind keys, unknown keys, and malformed values fail loudly as structural config errors.
- **SPEC-004.C43:** The effective approved-spool config is a shallow coordinate overlay of `spools.edn` followed by `spools.local.edn`; local entries intentionally replace shared entries with the same coordinate and normalized entries include source metadata for the winning file and `:kind`. Missing files yield an empty contribution unless user code explicitly requires spools to exist. Per-spool missing, unreadable, or add-spools-failed roots are sync outcomes rather than structural config errors.
- **SPEC-004.C44:** `skein.api.runtime.alpha/sync!` takes an explicit runtime, materializes approved git coordinates (SPEC-004.C91), and uses Clojure runtime dependency tooling to add each entry's effective root to the weaver runtime, recording per-spool loaded, already-available, or failed outcomes in weaver memory. Sync outcome maps are kind-shaped: every result carries `:kind` with only kind-relevant source fields (`:local/root` for local; `:git/url`/`:git/sha` plus `:git/tag`/`:deps/root` when configured for git), never nil-stuffed cross-kind keys and never manifest-derived data. Per-spool failures report fetch, root, and runtime-add/dependency-policy failures loudly in the sync result.
- **SPEC-004.C45:** Runtime spool availability and runtime activation are distinct. Making a root available allows weaver-side `require`; activation remains explicit trusted Clojure such as `use!`, direct function calls, or selected workspace-relative `load-file`.
- **SPEC-004.C46:** `skein.api.runtime.alpha/use!` takes an explicit runtime and records module-use attempts and loaded, skipped, or failed outcomes for weaver-lifetime introspection. `skein.api.runtime.alpha/reload!` takes an explicit runtime, clears weaver-lifetime approved-spool sync state, module-use state, named queries, views, patterns, lifecycle hooks, event handlers, queued events, and recent event failures, then reloads selected workspace startup files (`init.clj`, then `init.local.clj`) inside that weaver for common config hot-reload workflows. Missing startup files are skipped; present failing files throw with file context. Reload does not unload already-loaded Clojure namespaces or vars. This state is not durable package metadata.
- **SPEC-004.C47:** Runtime dependency loading is weaver-wide. There is no per-module version isolation or unloading guarantee; clean replacement may require weaver restart.
- **SPEC-004.C48@2:** Runtime source acquisition happens only at `sync!` for explicitly approved sha-pinned git coordinates. Skein must not silently clone repositories, add Git submodules, fetch source as part of module activation, or fetch anything a coordinate's own approval did not pin. No declared prerequisite ever triggers acquisition.
- **SPEC-004.C49@2:** Approved local roots and approved sha-pinned git coordinates are the only source kinds. Package registries, version ranges, dependency solving, lockfiles beyond approved git SHA pins, fetch-time code execution, and CLI package commands remain outside the contract. Maven artifact downloads for approved spool-root `deps.edn :deps` entries are allowed only under SPEC-004.C94a.
- **SPEC-004.C50:** Blessed `skein.api.*.alpha` namespaces are loaded from the mill-resolved Skein source checkout/classpath for the selected workspace. Startup or REPL use fails loudly if those namespaces are unavailable.
- **SPEC-004.C91:** Approved git coordinates materialize into a content-addressed cache at `<cache-base>/skein/spools/<sha>` (cache-base: `XDG_CACHE_HOME` when set and non-blank, else `~/.cache`); the effective root adds `:deps/root` when present. A non-empty cache path is a hit: no git invocation, no network, no tag re-verification, outcome `:fetch :cached`. A miss shells out to PATH-resolved system git, fetching the pinned sha into a temporary directory (shallow, with full-fetch fallback). If direct sha fetches fail, Skein performs one bounded fetch of advertised tags and branch refs from the same remote and then checks out the pinned sha from the refreshed temporary repository. Successful fetches are atomically renamed as content only (no `.git`) into place, outcome `:fetch :fetched`. Git command failures are per-spool `:failed`/`:fetch-failed` sync outcomes with exit code and trimmed stderr — never structural errors or exceptions out of `sync!`. When the bounded advertised-refs refetch still cannot fetch or verify the pinned sha, the failure outcome also includes the remote URL and cache path.
- **SPEC-004.C92:** When `:git/tag` is present and a fetch occurs, the tag (including the peeled `^{}` form for annotated tags) must resolve via `git ls-remote` to exactly the pinned sha before installation; mismatch or absence yields `:failed`/`:tag-mismatch` with expected/actual data and nothing cached.
- **SPEC-004.C93:** `sync!` does not parse `spool.edn`, validate manifest shape, compare manifest `:coordinate`, or compute manifest-declared prerequisite state. Spool prerequisites are documentation and consumer-owned approval/activation choices, not sync-time manifest data. Per-spool failures still report fetch, root, and runtime-add/dependency-policy failures loudly in the sync result.
- **SPEC-004.C94a@spool-contract:** A spool root's `deps.edn` top-level `:deps` is allowed for every approved spool kind (`:git` and `:local`) and for entries sourced from either `spools.edn` or `spools.local.edn`.
- **SPEC-004.C94a.1@spool-contract:** Every `:deps` entry must be a Maven coordinate map containing `:mvn/version`. Source-bearing tools.deps coordinates and paths, including `:git/url`, `:git/sha`, and `:local/root`, are not allowed inside a spool root's `deps.edn :deps`. Standard Maven coordinate refinement keys `:exclusions`, `:classifier`, and `:extension` are allowed alongside `:mvn/version`.
- **SPEC-004.C94a.2@spool-contract:** `sync!` validates this policy before calling `add-libs`. Any non-Maven `:deps` entry fails that spool's sync loudly as a per-spool runtime-add/dependency-policy failure before dependency resolution or classpath mutation for that entry. Mutable Maven versions are also forbidden: `:mvn/version` must not end in `-SNAPSHOT` and must not be the `RELEASE` or `LATEST` meta-version; violations use the same loud per-spool dependency-policy sync failure. A spool root `deps.edn` must not contain top-level `:mvn/repos` or `:mvn/local-repo`; repo redirection is not honored and fails that spool's sync loudly. Resolution uses only the weaver process's ambient Maven repository configuration.
- **SPEC-004.C94a.3@spool-contract:** Allowed Maven dependencies are resolved into the live weaver runtime during `sync!` through the same `clojure.repl.deps/add-libs` runtime dependency path used for spool roots. Transitive dependency semantics are plain tools.deps/add-libs Maven expansion of the declared coordinates. A spool root `deps.edn`'s `:aliases` and any other top-level key outside `:paths` and `:deps` — except the keys C94a.2 rejects loudly (`:mvn/repos`, `:mvn/local-repo`) — are ignored by `sync!`; no alias activation participates in the contract. Runtime dependency loading remains weaver-wide: there is no per-spool version isolation or unloading guarantee.
- **SPEC-004.C94:** `use!` no longer refuses activation because of manifest-declared needs or manifest-declared provides. The existing `use!` gates for `:after` ordering, target load, and `:call` remain otherwise unchanged. The `:spools` approval/sync-state gate is the blessed early prerequisite check: under `:required? true`, skip reasons `:not-approved`, `:not-synced`, and `:sync-failed` throw as required skipped activation. Check order: approval/sync state, `:after`, target load, `:call`.

## SPEC-004.P10 Runtime transformation primitives

- **SPEC-004.C51:** Skein ships blessed source-visible `skein.api.graph.alpha`, `skein.api.views.alpha`, and `skein.api.batch.alpha` namespaces for trusted runtime transformation workflows. They build on the runtime spool workspace model but are built-in namespaces, not user/community spools requiring `spools.edn` approval.
- **SPEC-004.C52:** `query-ids` returns a vector of strand ids for an ad hoc query definition or registered query name, ordered by the same stable strand ordering as `list` query results.
- **SPEC-004.C53:** `burn-by-id` and `burn-by-ids` physically delete strands and incident edges. `burn-by-ids` preserves first-occurrence input order with duplicates collapsed in its `:burned` result, returns `{:burned [] :count 0}` for empty input, and fails loudly if any requested id is missing.
- **SPEC-004.C53a:** `strands-by-ids` accepts a collection of strand ids, collapses duplicate ids by first occurrence, returns normalized strand rows in that first-occurrence input order, returns `[]` for empty input, and fails loudly if any requested id is missing.
- **SPEC-004.C54:** `ancestor-root-ids` traverses upward over one declared acyclic relation, defaulting to `parent-of`, where parent `from_strand_id` points to child `to_strand_id`. Seed ids are depth-zero candidates. With `:where`, it returns topmost matching ancestors on every path; without `:where`, it returns graph roots with no parent in the selected relation. Results are deduplicated and stable-sorted by id. Empty seed input returns `[]`; any missing seed id fails loudly. Traversal over undeclared annotation relations fails loudly.
- **SPEC-004.C55:** `subgraph` expands downward over one declared acyclic relation, defaulting to `parent-of`, from root ids and returns `{:root-ids [...] :strands [...] :edges [...]}`. The JSON socket accepts `{"root_ids":[...],"type":"relation"}` with `type` optional, and transports the same normalized graph shape as JSON. `:root-ids` preserves first-occurrence input order with duplicates collapsed. `:strands` contains normalized rows for roots and descendants, ordered by stable strand id. `:edges` contains only internal selected-relation edges connecting included strands, ordered by `from_strand_id`, `to_strand_id`, then edge type. Empty root input returns `{:root-ids [] :strands [] :edges []}`; any missing root id fails loudly. Traversal over undeclared annotation relations fails loudly.
- **SPEC-004.C55a:** `incoming-edges` and `outgoing-edges` resolve one strand-set's direct adjacency for a given edge type without graph traversal: `incoming-edges` returns edges whose `to_strand_id` is one of the ids (a strand's parents/annotators, indexed on `(to_strand_id, edge_type)`), and `outgoing-edges` returns edges whose `from_strand_id` is one of the ids (a strand's children, indexed on the `from_strand_id` primary-key prefix). Both take any valid edge type (annotation relations included — no acyclicity requirement) and return normalized `{:from_strand_id :to_strand_id :edge_type :attributes}` rows ordered by the queried side then the other. An empty id set returns `[]` without querying. Adjacency is deliberately lenient — unlike `subgraph`, `ancestor-root-ids`, and `strands-by-ids`, which fail loudly on a missing seed id, these edge-projection primitives treat an id absent from `strands` as indistinguishable from one with no matching edges (both contribute no rows), so callers needing a missing-id error validate at their own boundary.
- **SPEC-004.C56:** View registry entries are named by simple unqualified names and point to fully qualified Clojure function symbols resolvable in the weaver JVM. Duplicate registration replaces the prior entry for reload workflows.
- **SPEC-004.C57:** View invocation resolves the registered function symbol in the weaver JVM and calls it with one context map containing at least `:params`. View functions are read-only transformations in this feature; mutating workflows require a separate contract.
- **SPEC-004.C58:** View registry contents are weaver-lifetime runtime state and are not durable across restarts. Registry introspection returns serializable entries, not function objects.
- **SPEC-004.C59:** View registry weaver operation names are `:register-view!`, `:view!`, and `:views`.
- **SPEC-004.C59a:** Batch graph mutation uses the weaver operation name `:apply-batch`. It accepts the trusted batch payload contract and returns normalized Clojure data without requiring JSON socket exposure.
- **SPEC-004.C59b:** Pattern-created batches through `weave` and trusted transactional graph batch mutation through `apply-batch` are two doors into one transactional batch engine at different trust tiers: `weave` is the named, spec-checked, create-only CLI-safe door; raw batch remains trusted Clojure because it can create, update, burn, and upsert edges.
- **SPEC-004.C60:** Pattern registry entries are named by simple unqualified names and point to an optional non-blank doc string, fully qualified Clojure function symbol, and input spec name resolvable in the weaver JVM. Duplicate registration replaces the prior entry for reload workflows.
- **SPEC-004.C61:** Pattern invocation validates input against the registered spec before calling user code, invokes the function with `{:input input}`, requires a batch strand vector return value, and delegates persistence through the weaver API for id generation, ref resolution, edge insertion, cycle checks, transactionality, and lifecycle hook policy.
- **SPEC-004.C62:** Pattern explanation returns serializable guidance containing the pattern name, optional doc string, function symbol string, input spec string, printable spec form, and expanded schema details when available. This is caller guidance; invocation-time spec validation remains authoritative.
- **SPEC-004.C63:** Pattern registry contents are weaver-lifetime runtime state and are not durable across restarts. `skein.api.runtime.alpha/reload!` takes an explicit runtime, clears pattern state before loading selected config again.
- **SPEC-004.C63a:** CLI operation registry entries are named by simple unqualified names and carry a fully qualified Clojure handler function symbol plus an op metadata map: `:doc`, `:arg-spec` (blessed parser spec, optional for raw-envelope ops), `:stream?` (default false), `:deadline-class` (`:standard`/`:unbounded`; stream ops default `:unbounded`), and `:hook-class` (`:read`/`:mutating`, default `:mutating`); unknown metadata keys fail loudly. The registry records `:provenance` (the registering namespace, derived from the handler symbol) itself; caller-supplied provenance fails loudly. `register-op!` on an existing name fails loudly naming both registrants; deliberate override is explicit `replace-op!`, which requires the name to exist. `skein.api.runtime.alpha/reload!` clears operation state before re-running config, so reload workflows never trip the collision check.
- **SPEC-004.C63b:** CLI operation invocation resolves the registered function symbol and calls it with one context map containing `:op/name`, raw string vector `:op/argv`, `:op/runtime`, `:op/runtime-metadata`, `:op/payloads` (default `{}`), and any envelope-supplied `:op/cwd`/`:op/worktree-root`/`:op/git-common-dir`/`:op/timeout`; stream-class invocations additionally receive `:op/emit!`. When the entry declares `:arg-spec`, the blessed parser (`skein.api.cli.alpha`) runs first — resolving whole-value payload references and validating argv — and supplies the parsed map as `:op/args`; parse failures are loud domain errors and the handler is not called. Raw-envelope ops receive the unparsed context and own their argv/payload handling. The return value is transported as JSON-compatible data to `strand` callers. Operation handlers are trusted userland code and may intentionally block for workflow waits, agent awaits, or other long-running local orchestration — such ops must register `:deadline-class :unbounded` (SPEC-004.C26b).
- **SPEC-004.C63c:** The weaver installs exactly one built-in CLI operation before user config loads: `help` (`:hook-class :read`), a registry projection — no argv lists all entries (name, doc, provenance, stream?, deadline-class, hook-class) sorted by name; one positional op name returns that op's full detail including the parser rendering of its arg-spec or a raw-envelope marker; unknown names fail loudly with available names. `help` is registered through the public path, replaceable via `replace-op!`, and maskable like any op — unknown-op errors carry available names, so masking it yields no secrecy. `skein.api.runtime.alpha/reload!` takes an explicit runtime, clears operation state, reinstalls built-in operations, and then reloads selected config.
- **SPEC-004.C63d:** Arg-specs (SPEC-003.C60/C64/C65) integrate at three seams: `register-op!`/`replace-op!` run the parser's shared structural validator for both flat and subcommand arg-specs, failing loudly at registration on malformed containers, non-keyword arg names, unsupported `:type` values, invalid subcommand nesting, or reserved names; op invocation parses through the blessed parser before the handler runs, so argv errors are parse-phase domain errors and the handler is not called; and the built-in `help` op detail includes the parser `explain` projection.
- **SPEC-004.C63e:** Ops whose arg-spec declares `:subcommands` carry a help invocation alias: when the invocation's argv is exactly one token equal to `help`, `-h`, or `--help` and the envelope carries no payloads, the invocation returns the op's help detail projection (identical to `help <op>`, SPEC-004.C63c) as a normal successful single result. The alias resolves **before lifecycle hook gating** — a help-alias invocation is a read-class registry projection, so the target op's mutating-class hooks do not fire — and the handler is never called. Any other argv shape (extra tokens, attached payloads, the token in a non-sole position) is not an alias and flows through normal parsing and its loud errors; ops without `:subcommands` (flat arg-spec, raw-envelope) never trigger the alias. `help`/`-h`/`--help` are reserved subcommand names (SPEC-003.C65), so the alias shadows nothing.

## SPEC-004.P10a Event helpers

- **SPEC-004.C64:** Skein ships `skein.api.events.alpha` as a blessed source-visible explicit-runtime namespace for trusted config and live weaver REPL workflows. It exposes registration, unregistration, registry introspection, recent failure introspection, event submission, and a blocking event-lane quiescence await (SPEC-004.C74b).
- **SPEC-004.C65:** Event handler registration uses a stable key, a non-empty set of event type keywords, and a fully qualified function symbol resolvable in the weaver JVM under the runtime spool classloader. Duplicate keys replace prior entries for reload workflows.
- **SPEC-004.C66:** Event registry introspection returns data-first entries containing handler keys, subscribed event type sets, handler function symbols, and metadata. It does not expose function values or worker internals.
- **SPEC-004.C67:** Event handler failures are recorded in bounded weaver-lifetime introspection state. Handler exceptions do not fail the already-committed mutation that emitted the event.
- **SPEC-004.C68:** Event helper operations route directly inside the weaver JVM with an explicit runtime argument; connected-client workflows use `skein.repl` or trusted `skein.core.client` calls instead. They are not public JSON socket operations.
- **SPEC-004.C69:** Event registry contents are runtime state, not durable storage, and are reinstalled by trusted startup config, activated spools, or live in-weaver REPL forms after weaver restart.
- **SPEC-004.C70:** Runtime config reload stops event dispatch, discards queued events that have not started handling, clears event handlers, CLI operations, and recent handler failures, reinstalls built-in CLI operations, then re-runs selected workspace startup files (`init.clj`, then `init.local.clj`) before resuming event dispatch for the fully layered config. Reload does not unload Clojure namespaces or remove already-loaded vars.
- **SPEC-004.C71:** Events are Clojure maps with at least `:event/type`, `:event/id`, `:event/at`, and `:event/source`. Strand mutation events currently include `:strand/added`, `:strand/updated`, `:strand/superseded`, and `:strand/burned`.
- **SPEC-004.C72:** Add events include `:strand/id` and the normalized created `:strand`. Update events include `:strand/id`, submitted `:strand/patch` including any `:edges`, normalized `:strand/before`, and normalized `:strand/after`. Supersession events include `:strand/id`, `:strand/old-id`, `:strand/replacement-id`, normalized old-strand `:strand/before` and `:strand/after` rows, `:supersession/supersedes-edge`, and `:supersession/rewired-dependencies`. Burn events include `:strand/requested-ids`, `:strand/burned-ids`, and normalized pre-delete `:strand/before` rows.
- **SPEC-004.C73:** Events are emitted by Skein mutation operations after the database mutation succeeds, not by SQLite triggers or table watchers. Edge writes performed through `update` are represented by the same `:strand/updated` event in the MVP. Successful batch graph mutation emits `:batch/applied` first with final refs, created rows, updated before/after rows, burned ids/before rows, and edge outcomes. It then emits compatibility fanout in deterministic order: created strand events in result order, updated strand events in result order, and one aggregate `:strand/burned` event when any strands are burned. Compatibility fanout events include only a shared `:batch/id` for correlation; edge-only batch effects are represented by `:batch/applied` only.
- **SPEC-004.C74:** Event dispatch is asynchronous through a single sequential worker with a bounded queue. Enqueueing fails loudly when the queue is full. Recent handler failures are bounded to the latest 100 entries and include handler key, function symbol, event id/type, exception message, and time.
- **SPEC-004.C74a:** The `skein.spools.roster` reference spool is the shipped consumer example of this async event API: `install!` registers one `:strand/added`/`:strand/updated` handler that auto-stamps sufficient workflow/devflow graph roots into roster entries and refreshes their heartbeats on descendant mutations (SPEC-RosterSpool-001.C11/C14). It uses the C64–C74 contract and the C71–C72 mutation-event schema unchanged — reading only public strand attributes, never coupling the engine to the spool — so this integration adds no engine behavior; its contract lives at `spools/roster.md`.
- **SPEC-004.C74b:** `skein.api.events.alpha/await-quiescent!` `[runtime]` / `[runtime opts]` blocks until the single sequential event worker (SPEC-004.C74) has settled — the bounded queue is empty and no handler dispatch is in flight — then returns the runtime; on timeout it throws an `ex-info` (TEN-003), with a default budget from `skein.spools.test-support/await-budget-ms` overridable via `:timeout-ms`. "Idle" is an atomic dispatch-in-progress flag the worker raises *before* it claims an event and lowers after its handlers return, not a bare queue-empty check: because the worker claims an event before it could flip any flag, a naive queue-empty test could report settled while a just-claimed dispatch is still running. This is a lane-only primitive — it settles the shared event lane and observes no off-lane completion a handler may have started (the shell executor's `:shell` subprocess on a private worker pool, chime's per-dispatch notifier daemon threads, a subagent-executor gate's agent run reaching a terminal state), which keep their own completion signal layered on top and is never smuggled into `await-quiescent!`. No bare `quiescent?` predicate is exposed, because the useful operation is the await and a predicate invites the caller-rolled sleep loop this seam replaces.

## SPEC-004.P10b Lifecycle hooks

- **SPEC-004.C75:** Skein ships `skein.api.hooks.alpha` as a blessed source-visible explicit-runtime namespace for trusted config, activated spools, in-weaver code, and live weaver REPL workflows. Lifecycle hooks are synchronous weaver-side policy gates, separate from asynchronous post-commit events.
- **SPEC-004.C76:** Hook registry entries contain a stable key, a non-empty set of hook type keywords, a fully qualified function symbol resolved through the runtime spool classloader, an integer `:order` defaulting to `0`, and data-first metadata. Duplicate keys replace prior entries. Introspection returns entries in deterministic execution order, sorted by `:order` then key `pr-str`, and never exposes callable values.
- **SPEC-004.C77:** Runtime config reload clears lifecycle hook registry state with other weaver-lifetime registries, then reloads selected workspace startup files (`init.clj`, then `init.local.clj`); hook registrations are not persisted in SQLite.
- **SPEC-004.C78:** Validation hooks receive exactly one context map. Their return values are ignored; returning normally approves and throwing rejects. Transform hooks are explicit families and must return `{:hook/value replacement}`. Invalid transform return shapes or replacement values fail loudly.
- **SPEC-004.C79:** `:attributes/normalize` transform hooks thread JSON-compatible strand attribute maps before candidate validation for add, update patches, transactional batch created/updated strand entries, and pattern-produced create-only strand entries. They do not normalize edge attributes. Context includes at least `:hook/type`, `:hook/value`, `:request/source`, `:request/operation`, and `:mutation/operation`, plus strand-specific keys when known such as `:strand/id`, `:strand/ref`, `:strand/before`, `:strand/patch`, `:batch/ref`, and `:pattern/name`.
- **SPEC-004.C80:** `:payload/received` hooks gate socket `invoke` for ops whose registered `:hook-class` is `:mutating`; `:read` ops skip them, preserving the read-only exemption. Hooks run after JSON decode, protocol validation, identity verification, envelope-shape validation, and op resolution (unknown ops fail with the registry's available-names error *before* any hook runs), but before semantic dispatch. They may reject but not transform. Context includes at least `:hook/type`, `:request/source :json-socket`, `:request/operation :invoke`, `:request/id`, `:request/args` (the decoded envelope arguments), `:request/options`, and `:op/name` (the canonical resolved name).
- **SPEC-004.C81:** Pre-commit mutation hook families are `:strand/add-before-commit`, `:strand/update-before-commit`, `:strand/supersede-before-commit`, `:strand/burn-before-commit`, and `:batch/apply-before-commit`. They run synchronously before durable mutation commit where throwing aborts the mutation. Hook-rejected mutations commit no partial graph change and enqueue no mutation events. Context always includes `:hook/type`, `:request/source`, `:request/operation`, and `:mutation/operation`. Add hooks receive candidate created strand data with no before row. Update hooks receive strand id, patch, normalized before row, normalized after candidate row, and candidate edge operations requested by the update patch. Supersession hooks receive old id, replacement id, old before/after candidate rows, supersedes edge candidate, and dependency rewiring candidate data. Burn hooks receive requested ids and normalized before rows.
- **SPEC-004.C82:** Batch hooks use one common context schema for graph batch and pattern-created batch entry points: `:batch/source`, `:batch/payload`, `:batch/refs`, `:batch/created`, `:batch/updated`, `:batch/burned`, and `:batch/edge-ops`; source-specific empty entries are present as empty vectors or maps. For transactional graph batch mutation, `:batch/source` is `:apply`, `:batch/payload` is the submitted payload, and the remaining batch keys describe final refs, created candidates, updated before/after candidates, burned before rows, and candidate edge operations. For pattern-created batches, `:batch/source` is `:weave`, `:batch/payload` is the pattern-produced create-only batch in normalized batch shape, `:batch/updated` and `:batch/burned` are empty vectors, and context includes `:request/operation :weave`, `:pattern/name`, and `:pattern/input`; pattern-created batches pass through batch policy once.
- **SPEC-004.C83:** Successful hook-approved mutations preserve existing post-commit event behavior. Event handler exceptions still do not fail an already committed mutation. Trusted code that bypasses `skein.api.weaver.alpha` and calls lower-level persistence namespaces directly opts out of the blessed hook contract.
- **SPEC-004.C84:** Hook invocation failures propagate to callers as `ex-info` with top-level code `hook/failed`, including hook type, key, function symbol, original exception message/class, original `ex-info` data when present, and original `:code` as `:hook/cause-code` when present. JSON socket failures use the domain error envelope with code `hook/failed` and structured details.

## SPEC-004.P10c Local weaver peering

- **SPEC-004.C85:** Skein ships `skein.api.peers.alpha` as a blessed source-visible client-side namespace for trusted config, activated spools, and live weaver REPL workflows. It is loaded from the mill-resolved source checkout and is not a public JSON socket operation family; all of its helpers read published runtime metadata and speak to peer sockets from the calling process, so they behave identically inside the live weaver JVM and in connected client workflows.
- **SPEC-004.C86:** `peers` enumerates sibling weavers by reading `weaver.edn` metadata files (the Clojure-client artifact, SPEC-004.C11) under the mill state root (SPEC-004.C9a/C9b) and returns data-first rows including friendly name, selected workspace, weaver id, protocol version, socket path, state dir, and a running/stale determination. Present malformed metadata fails loudly per SPEC-004.C14; stale rows are reported as stale and are never callable peers.
- **SPEC-004.C87:** `peer` resolves exactly one running peer by logical friendly name or by explicitly path-like selected-workspace input (contains `/`, or starts with `~`); bare tokens always resolve as logical names, so a local directory named like a peer never shadows the name. A name matching more than one running weaver fails loudly listing every candidate with its workspace path; unknown or stale matches fail loudly. There is no auto-start: peer lifecycle belongs to the caller.
- **SPEC-004.C88:** `call!` invokes a named op on a resolved peer via the `invoke` envelope (or the minimal `status` operation) over the existing protocol: one request per connection with protocol-version and weaver-identity verification (SPEC-004.C13/C14/C23/C24). There is no client-side operation allowlist; rejection is receiving-side — unknown-op domain errors and the peer's `:payload/received` hooks remain the policy gate. Peering adds no socket operations or protocol fields; the receiving weaver cannot distinguish a peer from any other local client. `call!` supports single-result responses; a peer answering with a stream header fails loudly (`:peer/stream-unsupported`) without consuming the stream. Peer domain errors surface as structured client errors carrying peer identity and the peer's error envelope.
- **SPEC-004.C89:** Provenance for peer-initiated mutations is convention, not enforcement: peer callers identify themselves through existing freeform actor fields (for example workflow `complete!` `:by "weaver:<name>"`). The engine records nothing about caller identity beyond what operations already accept.
- **SPEC-004.C90:** Peering is same-machine, same-user only; SPEC-004.C32's exclusion of remote weaver access, authentication, and multi-user authorization is unchanged. Inter-weaver agreement is userland: the op registry (SPEC-004.C63a-c) is the contract surface, and versioning/introspection/deprecation conventions — version-suffixed dotted op names, a well-known describe op, deprecation stubs that fail loudly with structured replacement data and never pretend to succeed — belong to the `skein.spools.guild` reference spool contract, not the engine.

## SPEC-004.P10d Weaver scheduler

The weaver owns one scheduler subsystem for proactive durable wakeups: a substrate for the no-poller case, not a replacement for pull-based `wake-at` strand attributes plus views. It stores wake rows in dedicated SQLite tables, arms the next due wake inside runtime-owned state, and dispatches due handlers through the same serialized async mutation lane used by post-commit events (RFC-009). The blessed trusted-Clojure surface is `skein.api.scheduler.alpha` (SPEC-003.P4a).

- **SPEC-004.C97:** Scheduler state is runtime-owned, not module-level: it is acquired through `skein.api.runtime.alpha/spool-state` (SPEC-004.C95) with a declared version and a `:close-fn`, and it must not introduce a parallel module-level atom or bespoke reload registry. If the scheduler state shape changes, its spool-state version changes so stale executors/timers are closed and replaced under SPEC-004.C95/C96 reload semantics.
- **SPEC-004.C98:** A scheduler wake row contains at least a stable caller key, absolute wake instant, fully qualified handler symbol, persisted payload, status, attempt/fire/failure bookkeeping, and creation/update timestamps. Valid statuses cover pending, completed terminal history, cancelled, and failed; the exact table split is an implementation choice as long as pending and recent history/failure introspection stay data-first.
- **SPEC-004.C99:** Scheduling validates loudly before any durable write: the key must be a stable non-blank/non-nil value, `wake-at` must be an absolute instant shape the API accepts, the handler must be a fully qualified symbol resolvable in the weaver JVM/runtime classloader, and the payload encoding must be supported. Unsupported payload shapes and malformed timestamps are domain errors, never coerced defaults; a rejected schedule persists nothing.
- **SPEC-004.C100:** Startup and trusted config reload re-arm pending scheduler rows only after selected startup files finish loading, so handlers supplied by approved spools/config are resolvable before arming. Scheduler resources may initialize earlier, but durable-row handler resolution and timer arming are post-config steps. Re-arm cancels and cleans existing in-memory timers first and rebuilds from durable pending rows without leaking timer threads or double-arming rows. Runtime stop closes scheduler resources before storage closes via the spool-state close-on-stop lifecycle (SPEC-004.C91).
- **SPEC-004.C101:** Due detection is minimal: any pending wake with `wake-at <= now` is due and must be enqueued at least once (at-least-once delivery, not exactly-once — handlers must be idempotent). Malformed persisted rows fail loudly into failure introspection. Core does not implement cron, recurrence, retry/backoff, jitter, DST policy, missed-fire catch-up batching, or exactly-once semantics; a userland handler may schedule its own next wake.
- **SPEC-004.C102:** Scheduler handler invocation shares the existing serialized async dispatch worker by default, so clock-triggered handlers and post-commit event handlers observe one mutation lane rather than racing through independent worker pools. Queue-full is a transient delivery failure: it records an introspection failure but leaves or returns the durable wake to pending and re-arms it, so lane saturation cannot permanently drop a due wake before handler invocation. The durable delivery-attempt increment is generation-specific — it claims the exact wake generation selected (its key and wake instant) — so a wake cancelled or rescheduled in the race window between due-selection and the increment is never miscounted: a stale claim increments nothing, and a replacement generation's first real delivery still observes attempt 1.
- **SPEC-004.C102a:** Due-detection and arming read the current instant from the runtime clock component (SPEC-004.C1a), not a private scheduler clock: the scheduler owns no separate time authority. It registers its due-check as a runtime clock-consumer pump, so `skein.test.alpha/advance!` releases due wakes synchronously off an injected clock without waiting on the real timer thread. The durable rows, at-least-once delivery (SPEC-004.C101/C102), and the persisted `wake_at` millis are unchanged — the scheduler stays the durable wake primitive and merely learns the current instant from the shared clock. Its runtime state carries no clock slot, and that state-shape change bumps the scheduler spool-state version so SPEC-004.C95/C96 reload semantics close and replace stale executors/timers.
- **SPEC-004.C103:** A handler is a fully qualified symbol resolved in the runtime spool classloader and invoked with one context map containing the runtime, stable key, wake instant, attempt information, resolved handler symbol, and decoded payload. Return values are ignored. Handler-resolution failures and handler exceptions are captured in bounded recent failure/history data and never crash the weaver or the shared dispatch lane; idempotency remains the handler author's responsibility.
- **SPEC-004.C104:** The runtime exposes data-first scheduler introspection for pending wakes (ordered by wake-at ascending, so the earliest is the first pending wake), recent fires, recent cancellations, and recent failures. Recent history is bounded — the retention target is the latest 100 fires, cancellations, and failures per category, pruned during the write that records a new history entry. Introspection returns serializable maps and never exposes executor, timer, thread, or function objects.
- **SPEC-004.C105:** Scheduler tests must use deterministic clock/test seams. The runtime contract requires coverage for schema initialization, schedule/cancel/list persistence, startup re-arm, reload re-arm without duplicate fires, malformed schedule rejection, unresolvable handler failure capture, handler exception capture, and absence of scheduler rows from normal strand operations.

Design decisions: dedicated tables keep task queries, readiness, relation traversals, and graph burn/update semantics free of runtime coordination state while still giving the weaver enough durable state to re-arm after restart. The shared async lane is the default safety boundary — a separate scheduler worker would create a second mutation lane with different ordering/race behavior than post-commit events. There is no periodic safety tick in the contract: precise re-arm after scheduling, startup, and reload plus deterministic due-row tests are the arming mechanism, and a safety tick is added only if a concrete lost-arm window that precise re-arm cannot close is demonstrated.

## SPEC-004.P11 Removed compatibility

The weaver runtime and clients do not support compatibility lookup of old default `atom` workspaces, `tasks.sqlite`, `daemon.*` metadata, or `todo.*` / `atom.*` namespaces.
